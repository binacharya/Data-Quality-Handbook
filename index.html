🏗️

Imagine building a house on a shaky foundation

It doesn't matter how beautiful the walls, windows, or roof are — if the foundation is wrong, everything above it is at risk. Data is the foundation of every business decision, every dashboard, and every AI model. This handbook is about making sure that foundation is solid.

Over the last decade, companies have poured billions into cloud platforms, AI tools, and analytics systems. Yet one stubborn problem keeps getting in the way: the data feeding all of these systems is often wrong, incomplete, or inconsistent. Studies show that data professionals spend more than half their time cleaning data rather than using it. Bad data costs businesses an estimated $12.9 trillion every year worldwide.

Why does this keep happening?
🔌
Data comes from many places

Apps, spreadsheets, sensors, third-party feeds — each source has its own format, quirks, and quality issues. When they all connect, problems multiply.

👥
No single person owns it

Data passes through many teams — engineers, analysts, business users. Without clear ownership, quality checks fall through the cracks.

⏩
Speed beats accuracy

Teams under pressure to ship fast often skip validation steps. Problems that could have been caught early get buried deep in the pipeline.

🤖
AI makes it worse if ignored

AI systems trained on bad data don't just make mistakes — they make mistakes confidently, at scale, and automatically. The stakes are higher than ever.

The Data Quality Lifecycle

Think of data like a package being shipped across the country. At every stop — warehouse, truck, sorting centre, delivery van — something can go wrong. Data quality means checking the package at each stop, not just when it arrives. The six stages below are those stops.

Stage 1
Source Systems
Where data is born
🏭

Source systems are the apps, databases, and sensors that create data — things like your CRM, point-of-sale system, or a web form. Problems here are the cheapest to fix. If a customer types their email incorrectly and no one catches it, every downstream system will use the wrong email forever.

✔ Required fields filled in ✔ Values in allowed range ✔ Correct format (e.g. dates) ✔ No duplicate IDs
Stage 2
Data Ingestion
Moving data into your platform
🔄

Ingestion is the moment data travels from source systems into your data platform (e.g. a data warehouse or data lake). Like loading goods onto a truck, you need to check that nothing got lost in transit — especially if the source schema changes without warning.

✔ All expected records arrived ✔ Column names unchanged ✔ Data types match expectations ✔ No truncation or encoding errors
Stage 3
Raw / Bronze Layer
Your first copy of the data
🗄️

The raw layer stores an untouched copy of what arrived — exactly as the source sent it. Think of it as a secure archive. Before any cleaning or transformation, you check for obvious corruption: are all the files there? Is the row count what you expected? Are timestamps making sense?

✔ Expected row count arrived ✔ No file corruption ✔ Timestamps are plausible ✔ Null rates within tolerance
Stage 4
Transformation Layer
Cleaning and reshaping data
⚙️

Transformations clean, join, and reshape raw data into something useful — like a chef turning raw ingredients into a dish. But this is where business logic gets applied, and business logic can be wrong. A mis-coded rule here can silently corrupt every report that depends on this layer.

✔ Business rules applied correctly ✔ Joins produce expected output ✔ Aggregations add up correctly ✔ No unexpected NULLs introduced
Stage 5
Analytics Layer
The data your teams actually use
📊

This is what powers dashboards, reports, and KPIs. Executives make decisions based on this layer. If the revenue figure on a Monday morning dashboard is wrong, trust erodes immediately. Quality checks here focus on consistency — does today's number look right compared to last week?

✔ KPIs consistent day-to-day ✔ No metric spikes or drops ✔ Totals match across reports ✔ Data is fresh enough for decisions
Stage 6
AI & ML Models
The most sensitive data consumer
🤖

Machine learning models are extremely sensitive to data quality. A model trained on clean data but fed messy data at run-time will produce confident but wrong predictions. This is called training-serving skew — the silent killer of AI projects. DQ checks here focus on drift: has the data changed since the model was trained?

✔ Training & serving data match ✔ Feature distributions stable ✔ No new unseen categories ✔ Model outputs monitored over time
🔄
It's a Loop, Not a One-Way Street

When a quality issue is discovered at the analytics layer, the investigation works backwards through the pipeline — transform logic, raw data, ingestion rules, source system. Good data quality practices leave a trail of checks at every stage so that root cause analysis takes minutes, not days.

Glossary of
Key Terms

📖

New to data? Don't worry. Every field has its own language, and data is no different. This glossary explains the most common terms you'll encounter — with a plain-English translation for each one so you always know exactly what people mean.

The Six Quality Dimensions
Accuracy
The degree to which data correctly represents the real-world entity or event it describes.

Is the value actually true? E.g. does the customer's name in the system match their real name?

Completeness
The proportion of required data fields that are populated with values versus null or missing.

Are all the fields filled in? A form with a blank phone number field is incomplete data.

Consistency
Agreement of data values across different systems, datasets, or points in time.

Does the same customer appear with the same name and address in every system?

Timeliness
Whether data is available when needed and represents the correct point in time for its intended use.

Is the data fresh enough? Yesterday's stock price is useless if you need today's.

Validity
Conformance of data values to defined formats, ranges, types, or referential constraints.

Does the value follow the rules? E.g. a date field should contain a real date, not "N/A".

Uniqueness
The absence of duplicate records for a given entity or identifier within a dataset.

Is each customer listed only once? Duplicates cause double-counting in reports.

Pipeline & Data Flow
Data Pipeline
An automated series of processes that move, transform, and load data from one system to another.

Think of it like an assembly line — data enters one end raw and comes out the other end ready to use.

ETL / ELT
Extract, Transform, Load / Extract, Load, Transform — patterns for moving data between systems.

How data gets from A to B. ETL cleans it on the way; ELT cleans it after it lands.

Data Lineage
The end-to-end record of how data flows from its origin through transformations to its final destination.

A paper trail for your data — where did it come from, what happened to it, and where did it go?

Schema Drift
Unexpected changes to the structure of a dataset (column additions, type changes) that break downstream processes.

Imagine a spreadsheet that suddenly has a new column your code doesn't know about — things break.

Data Drift
The gradual change in the statistical distribution of a dataset over time, which can degrade model performance.

The data slowly changes shape over months. AI models trained on old data start giving worse answers.

Freshness
How recently data was updated or loaded, indicating whether it reflects current real-world state.

Like checking the "best before" date on food — stale data can lead to bad decisions.

People & Process
Data Steward
A business-side owner responsible for ensuring the quality, governance, and appropriate use of data within a domain.

The person who "owns" a dataset and is accountable for keeping it clean and well-documented.

Data Producer
The person, team, or system responsible for creating or sourcing data that is consumed downstream.

Whoever creates the data — e.g. the sales team that fills in customer records.

Business Rule
A condition or constraint derived from business logic that data must satisfy to be considered valid.

A rule like "order amount must be greater than zero" — defined by the business, not just the database.

SLA (Service Level Agreement)
A commitment defining expected data availability, freshness, and quality targets between teams.

A promise between teams — e.g. "this dataset will be updated and ready by 6 AM every day".

Data Contract
A formal agreement between a data producer and consumer specifying the expected schema, quality standards, and SLAs.

A written agreement: "I'll send you data in this exact format, and it will always meet these standards."

KPI (Key Performance Indicator)
A measurable metric used to evaluate success against defined objectives.

A number that tells you how well something is working — like "% of records with no missing values".

Tools & Monitoring
Data Catalog
A metadata repository that inventories an organisation's data assets, enabling discovery and governance.

Like a library index for your data — helps people find what data exists and understand what it means.

Metadata
Data about data — including schema, lineage, statistics, and documentation.

The label on the tin. It doesn't contain the actual data, but tells you what's inside and where it came from.

Data Observability
The ability to fully understand the state and health of data in a system through monitoring and telemetry.

Like a health dashboard for your data — you can see at a glance if something is wrong, before users notice.

Anomaly Detection
Automated identification of data points or patterns that deviate significantly from expected norms.

Automatically spotting things that look "off" — e.g. revenue that suddenly drops to zero on a Tuesday.

Null Rate
The percentage of missing or null values in a column; a key indicator of completeness.

If 30% of your "email" column is blank, you have a 30% null rate — a sign something went wrong at input.

dbt (Data Build Tool)
An open-source transformation framework for SQL-based data pipelines that supports testing and documentation.

A popular tool for cleaning and organising data inside your warehouse — with built-in quality tests.

Core Data Concepts
Golden Record
The authoritative, deduplicated, and reconciled version of an entity (customer, product) across systems.

The single "source of truth" for a customer or product — the one record everyone agrees is correct.

Deduplication
The process of identifying and removing duplicate records from a dataset.

Removing copies. If "John Smith" appears three times in your customer list, deduplication leaves just one.

Snowflake
A cloud-based data warehousing platform widely used for data storage, transformation, and analytics.

A cloud database service — think of it as a very large, fast, shared filing cabinet for business data.

Chapter 01 · Foundations

What is
Data Quality?

🗺️

Imagine your GPS gives you wrong directions

You trust it, you follow it, and you end up at the wrong place. That's exactly what bad data does to a business. Decisions get made — confidently — based on information that's wrong, incomplete, or out of date. Data Quality is the discipline of making sure your data can be trusted to guide you to the right place.

🎯
Definition
Data Quality (DQ)
Data quality means your data is fit for its purpose. It is accurate enough, complete enough, fresh enough, and consistent enough for the decisions and processes that rely on it. Quality is not a fixed standard — it depends on what the data is used for. The same data can be high quality for one use case and dangerously low quality for another.
✅ Accurate 📋 Complete 🔄 Consistent ⏰ Timely 🔢 Valid 🔍 Unique

Quality Depends on Context

Before you can judge whether data is "good", you need to answer four questions:

1
🎯
What is this data used for?
Customer phone numbers matter a lot for delivery notifications, but very little for analysing average order value. Quality requirements follow the use case.
2
👥
Who relies on it?
A data scientist exploring trends can tolerate some missing values. A pharmacist checking allergies cannot. The audience determines the acceptable error rate.
3
⚖️
What decisions does it inform?
Low-stakes decisions (what colour to A/B test) can survive imperfect data. High-stakes decisions (loan approvals, medical dosing) cannot. Stakes raise the quality bar.
4
💥
What happens when it's wrong?
If wrong data causes a minor inconvenience, quality requirements are flexible. If it causes a regulatory fine, a lost sale, or patient harm — quality becomes non-negotiable.

The 4 Types of Data Quality Problems

Most data quality issues fall into one of these four categories. Recognising the type helps you know where to look and how to fix it:

Critical
Content Problems — wrong or missing values
The actual data values are wrong, empty, or make no sense. This is the most dangerous type because it looks fine on the surface.
Example: A patient's age is recorded as 312. A product price is NULL. An email address is "test@test".
High
Structural Problems — wrong shape or format
The data values might be correct, but they are stored in the wrong format or structure — breaking systems that try to read them.
Example: A date column stores "January 5" instead of "2025-01-05". A number column stores "£12.50" as text.
High
Consistency Problems — same thing, different answer
The same piece of information appears differently across two systems. Both can't be right — but neither flags an error on its own.
Example: The CRM says a customer's city is "London". The billing system says "LONDON". The warehouse says "london".
Medium
Timeliness Problems — right data, wrong time
The data was correct when it was recorded, but it's now too old to be used for time-sensitive decisions. It's outdated, not wrong.
Example: Inventory data is 6 hours old. A customer's address hasn't been updated since 2019. A price feed is from yesterday.

The 1-10-100 Rule — Why Early Detection Saves Money

Research by IBM shows a consistent pattern: the later you catch a data error, the more expensive it is to fix. This is called the 1-10-100 Rule:

🔍
$1
Caught at source
A validation rule flags the wrong value before it enters the system. One person fixes it in seconds.
📊
$10
Caught in a report
An analyst spots the error in a weekly dashboard. Hours of investigation, data reload, and re-publishing are required.
💥
$100
After a decision is made
A business decision was made based on wrong data. Reversing it costs money, damages trust, and may trigger compliance review.

Why Data Quality Degrades Over Time

Even clean data gets messy. Here are the six most common reasons data quality slowly deteriorates — often without anyone noticing:

👥
Organisational changes

New teams enter data differently. Definitions shift. What "active customer" meant in 2022 might mean something different today — but no one updated the documentation.

🔄
System migrations

Moving data between systems (old CRM → new CRM, on-premise → cloud) introduces mapping errors. Fields get renamed, values get mis-translated, and records go missing.

🏗️
Schema evolution

A source system adds a new column or renames an existing one. The downstream pipeline doesn't know — so the new column arrives as NULL or breaks the pipeline entirely.

📈
Volume growth

When data was small, a 0.1% error rate was barely noticeable. When you have 50 million records, that's 50,000 bad rows affecting every report that touches them.

✍️
Manual processes

Humans make mistakes. Any process that involves typing data by hand has a 1–5% error rate as a baseline — no matter how careful the person is.

🔗
Integration sprawl

Every new data source is a new failure point. More APIs, more pipelines, more places for things to silently go wrong — especially without monitoring in place.

Chapter 02 · Foundations

The Six Dimensions
of Data Quality

The most widely adopted framework for measuring data quality uses six core dimensions. Originally formalized by DAMA International (Data Management Association) and grounded in the seminal 1996 academic research by Richard Wang and Diane Strong — "Beyond Accuracy: What Data Quality Means to Data Consumers" — these dimensions provide a structured vocabulary for describing, measuring, and communicating data quality across an organization.

✅

1. Completeness

All required data is present. Missing values silently break analytics and operational processes.

🎯

2. Accuracy

Data values correctly reflect the real-world entities they represent. Inaccurate data drives wrong decisions.

🔄

3. Consistency

The same data appears identically across all systems. Conflicting records erode trust in reporting.

⏰

4. Timeliness

Data is available within the required time window. Stale data is dangerous in operational settings.

🔢

5. Validity

Data conforms to defined formats, value ranges, and referential integrity rules. Invalid data causes failures.

🔍

6. Uniqueness

No duplicate records for entities that should appear only once. Duplicates distort metrics and create chaos.

Figure 2.1 — The Six Core Dimensions of Data Quality, as defined by DAMA International. Every DQ program must measure all six.

Deep Dive: Each Dimension Explained

The following sections explore each dimension in depth — what it means, why it matters, and real-world examples of what happens when it fails.

✅
Dimension 01
Completeness
92% Industry Avg
What it means: Completeness measures whether all required data fields contain values. A dataset is complete when no expected data is missing — no blank fields, no null values where values are mandatory, and no records absent from a population that should be fully represented. Completeness is often the first dimension organizations begin tracking because it is the most straightforward to detect.
🗂️ Real-World Scenarios
🛍️
E-Commerce / Retail

An online retailer's customer table has 18% of records missing phone numbers. When an order is delayed, the fulfilment team cannot contact customers — leading to failed deliveries, refund requests, and negative reviews. The missing data was traced to a mobile checkout flow that made phone number optional.

🏥
Healthcare

A hospital's patient intake form has a "known allergies" field frequently left blank — sometimes because the patient has no allergies, but often because the field was skipped. Without a mandatory distinction between "no allergies" and "not recorded", clinicians cannot safely prescribe medication.

📊
Finance / Reporting

A quarterly revenue report excludes 7% of transactions because the "cost_centre" field was not populated at point-of-entry. The CFO presents understated departmental costs to the board. The incompleteness is not noticed until audit — three months after the report was submitted.

📐 The Formula

Completeness % = (Populated fields / Expected fields) × 100
If a customer table has 10,000 rows and 800 have a NULL email address, email completeness = 92%. Most organizations define a minimum acceptable threshold per field — critical fields like ID and email often require 99%+.

❌ Poor Completeness

Customer records with NULL email addresses → email campaigns fail silently. 12,000 emails undelivered; marketing spend wasted; zero open rate attributed to poor content rather than missing data.

✅ Good Completeness

Mandatory validation at data entry enforces email capture. New records: 99.4% email completeness. A daily completeness check flags any record below 95% before it reaches the warehouse.

⚡
Business Impact IBM's 1-10-100 Rule: it costs $1 to prevent a data error, $10 to correct it after entry, and $100+ to fix the downstream consequences. Incomplete data reaching production systems multiplies this cost exponentially — a single missing field in a loan application can delay processing by days and trigger regulatory non-compliance.
🎯
Dimension 02
Accuracy
85% Industry Avg
What it means: Accuracy measures how closely data values match the true, real-world values they are intended to represent. Accurate data reflects reality — the correct name, the right address, the actual transaction amount. Inaccuracy means the data exists but is wrong. This is often harder to detect than missing data, because inaccurate records pass all presence checks and look perfectly valid on the surface.
🗂️ Real-World Scenarios
📦
Supply Chain / Logistics

A supplier's catalog lists a product weight as 2.5 kg. The actual weight is 4.2 kg. The logistics system calculates shipping costs on the incorrect weight — every shipment is under-charged, costing the company $180,000 annually before the error is found during a carrier audit two years later.

🏦
Banking / KYC

A customer's date of birth was entered as 1986 instead of 1968 — a transposition error during onboarding. The bank's age-verification rules approve the customer for a financial product they are ineligible for. The error surfaces only during a regulatory audit 18 months later, triggering a remediation program.

📈
Sales Analytics

Sales data shows Region A outperforming Region B by 23%. A currency conversion error is discovered: Region A figures are in USD, Region B figures are in GBP — making the comparison meaningless. Strategic headcount decisions had already been approved based on the false ranking.

Figure 2.2 — A single transposition error at data entry propagates an incorrect eligibility decision to every downstream system.

⚡
Business Impact Accuracy is the most trust-critical dimension. When executives lose confidence in data accuracy, they stop using dashboards and revert to gut instinct — a phenomenon called "spreadsheet proliferation." Gartner estimates poor data quality costs organizations an average of $12.9 million per year, with inaccuracy as the leading root cause.
🔄
Dimension 03
Consistency
78% Industry Avg
What it means: Consistency ensures that the same data — representing the same real-world entity — has the same value across all systems where it is stored or referenced. When the same customer appears differently in the CRM, ERP, and data warehouse, you cannot reliably join or reconcile those records. This creates a "multiple versions of truth" problem that undermines every cross-system report and integration.
🗂️ Real-World Scenarios
👤
CRM / Customer Data

A customer "Robert Johnson" appears as "Bob Johnson" in the CRM, "R. Johnson" in the billing system, and "JOHNSON, ROBERT" in the legacy database. The marketing team cannot build a unified customer profile — the same person receives duplicate offers, conflicting account summaries, and three separate welcome emails after a system migration.

💰
Finance / Reconciliation

The finance ERP shows a supplier invoice as "Paid — $42,300". The accounts payable ledger shows the same invoice as "Pending — $42,300". Two systems, same invoice, conflicting statuses. The inconsistency was caused by a silent batch sync failure. The supplier is chased for payment they had already received, damaging the business relationship.

🏪
Retail / Product Data

A product is named "Wireless Headphones Pro X" in the inventory system but "BT Headphone Pro" in the e-commerce catalog. A customer availability lookup fails because the names do not match — the item shows as "Out of Stock" even with 400 units in the warehouse. Lost sales accumulate for weeks before the discrepancy is found.

Figure 2.3 — The same customer exists in 4 systems with 4 different name and email formats. Only the CRM holds the authoritative value. Automatic record matching fails.

❌ Poor Consistency

Three business units report different quarterly revenue totals — $4.2M, $4.5M, and $4.1M — because each applies different rules for when revenue is "recognised". Boardroom debates about which number is correct delay strategic decisions by weeks.

✅ Good Consistency

A single "Golden Record" master data management system defines canonical customer, product, and revenue definitions. All downstream systems pull from this source. Reconciliation jobs run hourly; any deviation > 0.5% triggers an automated alert to the data steward.

⚡
Business Impact Consistency scores the lowest industry average (78%) of all six dimensions — because fixing it requires cross-system coordination, which is organizationally difficult. The Target Canada collapse ($2B+ CAD write-down) was largely driven by product master data inconsistency: the same product had different names, units, and codes across supplier, warehouse, and retail systems.
⏰
Dimension 04
Timeliness
88% Industry Avg
What it means: Timeliness measures whether data is available at the time it is needed for its intended use. Data can be complete, accurate, and consistent — yet still be "bad" if it arrives after the decision window has closed. Timeliness is always relative to context: what is "timely" for monthly reporting may be catastrophically late for fraud detection or inventory management.
🗂️ Real-World Scenarios
🚨
Fraud Detection / Banking

A bank's fraud model ingests transaction data with a 4-hour delay from batch ETL processing. By the time the model flags suspicious card activity, the fraudster has made 12 additional transactions across 3 countries. Real-time streaming would have triggered a block within 90 seconds of the first anomaly.

✈️
Aviation / Seat Inventory

An airline's seat availability system updates every 20 minutes. During a high-demand flash sale, 400 customers purchase the same 200 remaining seats because the inventory does not reflect bookings made in the previous 18 minutes. The airline must involuntarily deny boarding to 200 passengers and pay compensation.

🏬
Retail / Inventory

Stock levels are updated once nightly via a batch job. The online store shows a bestselling product as "In Stock" at 9 AM when it actually sold out at 11 PM the previous evening. 340 customers complete purchases, only to receive cancellation emails the next morning. Brand trust erodes.

Figure 2.4 — Fraud detection requires data within seconds. A 4-hour batch pipeline misses the intervention window entirely — allowing the fraud to continue unchecked.

❌ Poor Timeliness

Daily batch inventory sync. Online store shows "In Stock" 18 hours after last unit was sold. 340 customers place orders for an item that doesn't exist. Cancellation emails damage brand trust; refund processing exceeds original revenue from those orders.

✅ Good Timeliness

Event-driven architecture streams stock changes to the catalog within 2 seconds of each warehouse transaction. When the last unit is sold, the product page switches to "Out of Stock" automatically. Customer experience protected; oversell incidents near zero.

⚡
Business Impact Timeliness requirements vary enormously — from milliseconds (financial trading), to seconds (fraud, inventory), to hours (daily reporting), to days (regulatory filings). A key DQ practice is defining explicit freshness SLAs per use case, and building automated alerts when pipelines fail to meet those SLAs. The same data can be "timely" for one use case and dangerously stale for another.
🔢
Dimension 05
Validity
95% Industry Avg
What it means: Validity checks whether data conforms to predefined rules — including format constraints, permitted value ranges, referential integrity, and business logic. Data can be present (complete) and even factually correct (accurate), yet still be "invalid" if it violates the expected structure. Validity failures are caught through data contracts, schema validation, and constraint checking at the point of ingestion.
🗂️ Real-World Scenarios
📅
Healthcare / Insurance Claims

A patient's discharge date of "02/30/2024" passes basic text validation but is an impossible date. The record is accepted into the billing system, which generates a claim for February 30th — a date that does not exist. The insurer rejects the claim; payment is delayed by 45 days and staff must manually investigate and refile.

🌍
Logistics / Addressing

A shipping form accepts "ABCDE" as a valid US postal code — the field accepts any 5 characters, but US ZIP codes must be numeric. 2,800 parcels are routed to the wrong distribution center because the ZIP is invalid. The system had no referential check against a valid ZIP code lookup table.

📊
Analytics / Data Pipelines

Age of AI

🤖 The AI Data Quality Imperative

Every AI model is only as good as the data it was trained on. This is not a metaphor — it is a mathematical truth. Biased training data produces biased models. Incomplete training data produces models with blind spots. Inconsistent training data produces models that behave unpredictably. In the era of AI-driven business decisions, data quality is not a back-office concern — it is a board-level risk.

How Poor Data Quality Manifests in AI

The Golden Rule of AI: Garbage In, Garbage Out (GIGO)
Think of training an AI model like teaching a student from a textbook. If the textbook has wrong facts, missing chapters, and repeated pages — the student will learn those mistakes perfectly. AI doesn't question its data. It doesn't know when something is wrong. It will confidently repeat every error in its training data, at scale, and at speed. A human analyst might catch a suspicious number. An AI model will build its entire world view around it.

Here are the six most common ways poor data quality breaks AI systems — explained simply, with real examples:

⚖️
1. Training Bias
The AI learns prejudice from skewed data
When training data over-represents one group and under-represents another, the AI learns to favour the majority group — not because it's right, but because that's all it has seen.
Think of it like: Teaching someone to identify fruit by only showing them green apples. They'll call every apple "green" — not because all apples are green, but because that's all they were taught.
Real Example — Hiring AI Tool
Training Data (Biased)
92% of historical "successful hire" records are male engineers from 3 universities
What Should Happen
Diverse pool of successful hires across gender, background, and institution
💥The AI systematically ranks female candidates and non-target-university applicants lower — not due to skill, but because its training data encoded historical hiring bias as "the right answer."
📉
2. Concept Drift
The world changes; the model doesn't know
AI models are trained on historical data. When real-world patterns change — due to market shifts, events, or new customer behaviours — the model keeps using the old rules. It doesn't notice the world has moved on.
Think of it like: A weather model trained only on summers being asked to forecast in winter. It will keep predicting sunshine because it has never seen snow.
Real Example — Fraud Detection (COVID-19)
Pre-COVID Training Data
Buying groceries online = suspicious. Most fraud happened via online channels in 2019.
Post-COVID Reality
Online grocery orders became the norm for millions. Behaviour changed completely.
💥Banks using pre-2020 fraud models flagged millions of legitimate online grocery and delivery transactions as fraud in 2020–21 — blocking valid customers and generating thousands of false alerts daily.
🔇
3. Signal Noise from Duplicates
Repeated data drowns out real patterns
Duplicate records, outliers, and inconsistent values create "noise" — false signals that the AI mistakes for real patterns. The model wastes its learning capacity on noise instead of genuine insight.
Think of it like: Asking 10 people to rate a restaurant, but 7 of them are the same person who submitted multiple reviews. The average is meaningless — one person's opinion is disguised as many.
Real Example — E-commerce Recommendation Engine
Noisy Data
Product "Blue Sneakers" appears 3× due to duplicate SKUs: SKU-441, SKU-441B, sku_441. Combined, it looks like the most popular item.
Clean Data
One SKU per product. True popularity based on actual distinct customer purchases.
💥The recommendation engine promotes "Blue Sneakers" to every user regardless of their preferences, suppressing genuinely relevant products. Click-through rate drops 34%. The team blames the algorithm — the real problem is the data.
🕳️
4. Missing Context
NULL values teach the AI that "nothing" means something
When fields are missing (NULL), the AI doesn't skip them — it assigns them a default value (usually 0 or "unknown") and learns from that. If a pattern happens to correlate with NULLs, the model will use "missing data" as a decision signal — which is almost never what you want.
Think of it like: A doctor predicting health outcomes but using "no data" as if it means "healthy." A patient with no recorded allergies is very different from a patient who has no allergies — but a model trained on NULL = 0 won't know the difference.
Real Example — Hospital Readmission Predictor
Missing Data in Training
age=NULL, medications=NULL for 40% of records (older patients pre-digitisation). Model learns NULL age → low readmission risk.
Reality
NULL values are concentrated in elderly patients — who actually have the highest readmission risk.
💥The model systematically under-predicts readmission risk for elderly patients — the exact group who need the most intervention. Discharge planning fails for the highest-risk patients because the AI labelled them "low risk."
🔓
5. Label Leakage
The model accidentally "cheats" using future information
Label leakage happens when the training data accidentally includes information that would only be known after the event you're trying to predict. The model appears to perform brilliantly in testing — but fails completely in production because the "cheat" information doesn't exist at prediction time.
Think of it like: Giving students the answer sheet during a practice exam. They all score 100%. On the real exam with no answer sheet, they fail — because they memorised answers, not the subject.
Real Example — Customer Churn Prediction
Leaked Training Data
Features: last_login, plan_type, cancellation_date (future!), support_tickets. Model gets 99% accuracy.
Correct Features Only
last_login, plan_type, support_tickets, usage_30d. No future data. Realistic 78% accuracy.
💥The team celebrates a "99% accurate churn model" and launches it. In production, it predicts randomly because the leaking column doesn't exist for active customers. Months of model development wasted. The data quality error was invisible until deployment.
📡
6. Distribution Shift
Training population ≠ production population
A model trained on one type of data is deployed to score a different type. The characteristics (distributions) of real users are different from the training set, so every prediction is being made outside the range of what the model has learned.
Think of it like: Learning to drive on empty country roads, then being told to drive in central London on your first day. The skill was learned in completely different conditions from where it needs to work.
Real Example — Credit Scoring App
Training Data (Desktop Users)
Avg age: 42 · Income: £55K · Sessions: 2/month · Device: desktop
Production Data (Mobile-First Users)
Avg age: 26 · Income: £28K · Sessions: 14/month · Device: mobile
💥The credit model systematically assigns low scores to young mobile-first users — not because they're higher risk, but because they look nothing like the desktop users the model was trained on. A whole customer segment is unfairly denied credit.
⚠️ The Compounding Problem

Most AI failures are not caused by a single data quality issue — they are caused by multiple problems compounding. A model may have training bias and missing context and concept drift all at once. Each issue multiplies the others. This is why data quality for AI must be treated as a systematic discipline, not a one-time clean-up task before model training.

The Data Quality Pyramid for AI

What is the DQ Pyramid for AI? Think of building an AI model like constructing a house. You cannot build the roof before the walls, and you cannot build the walls without a solid foundation. Similarly, you cannot have "model-ready" features if the underlying data is incomplete, inconsistent, or unvalidated. Each layer of the pyramid must be achieved before the one above it can be trusted. Skipping a layer doesn't save time — it just hides a problem that will surface later in production.
1
🗄️
Complete & Accessible — The Foundation
In plain English: Your data actually exists, it can be read by your systems, and the most important fields are not empty. This sounds obvious — but it is the most commonly violated layer. An AI cannot be trained on data that doesn't exist or can't be accessed.
Why it matters for AI: If 30% of customer age fields are NULL, the model either learns that NULL is a valid age group, or it drops those rows entirely — potentially losing patterns that only appear in younger or older users.
💥 If you skip this: Your model trains on a fraction of the real population. The gaps aren't random — they're systematic — and the model learns the wrong reality.
✓ No NULL in critical fields✓ Freshness within SLA✓ All source tables ingested✓ Row counts match expectations

🔧
Structurally Valid — Correct Shape and Format
In plain English: Every field contains the right type of value in the right format. A date field holds actual dates (not "N/A" or "12-31-99"). A price field holds numbers (not "$12.99" with a dollar sign). A country code field holds "GB" (not "United Kingdom" or "uk" or "Britain").
Why it matters for AI: AI models work with numbers. When a model encounters "twelve" instead of 12, it either crashes, ignores the row, or encodes the text string as a meaningless code. Any of these outcomes corrupts the model's understanding of that feature.
💥 A pricing model trained on a "revenue" column that sometimes contains text values like "N/A" or "—" will learn that these strings predict something — they don't. They're just missing data.

In plain English: The same customer, product, or event is recorded the same way everywhere in your system. "Jane Smith", "J. Smith", and "jane smith" are the same person — but your AI doesn't know that unless you've done deduplication. If they're treated as three separate people, the model divides that person's behaviour into three weaker signals.
Why it matters for AI: Duplicates artificially inflate the weight of certain records. If customer 447 appears three times in training data, the model thinks their behaviour pattern is three times more common than it actually is. This skews every prediction involving similar customers.
💥 A churn model trained on duplicated customer records inflates the apparent size of certain segments. When deployed on clean production data, the predictions become inconsistent because the "population" the model learned from doesn't match reality.

In plain English: A subject matter expert — someone who actually works with this data — has confirmed it makes sense in the real world. Technical checks can verify that a credit score is a number between 300 and 850. But only a credit analyst can confirm that a score of 310 for someone with 15 years of pristine history is suspicious and probably a data entry error.
Why it matters for AI: Many data errors are technically valid but factually wrong. A patient listed as 7 feet 4 inches tall, a transaction of $0.01 for a commercial property sale, a product weight of 0.001g for a TV — these pass all format checks but are clearly wrong. Only a human expert can catch these, and the AI will learn from them as if they were real.

This is the layer where data scientists spend the most time — and it's wasted if the layers below are broken.
Why it matters for AI: AI models are mathematical functions. They require inputs within expected ranges, with consistent scales, and without surprise values they've never seen before. LLMs and Retrieval-Augmented Generation (RAG)

🤖 New to LLMs and RAG? Start Here.
Large Language Models (LLMs) — like ChatGPT or Claude — are AI systems trained on massive amounts of text. They can write, answer questions, summarise documents, and hold conversations. But they have a key limitation: their knowledge is frozen at the time they were trained. Ask an LLM about something that happened last week, and it won't know.

Retrieval-Augmented Generation (RAG) solves this by giving the LLM access to a live, searchable knowledge base. Instead of relying only on what it was trained on, the LLM first retrieves relevant documents from your organisation's knowledge base (policy docs, product manuals, customer data, research papers), then generates an answer using those documents as context.

Think of it like this: without RAG, the LLM is a brilliant person who read every book up to 2023 but is locked in a room with no internet. With RAG, they now have a filing cabinet in the room — and your answers are only as good as what's in that filing cabinet.
📅
Outdated Documents
The knowledge base contains old versions of documents that have since been updated. The retriever fetches the outdated version and the LLM answers based on old information — confidently and incorrectly.
Like asking someone to check the company handbook, but they're reading the 2019 edition when the 2025 edition has completely different policies.
Real impact: An HR chatbot built on company policy documents answers questions about parental leave using a 2021 policy that was updated in 2024. Employees receive incorrect information — and the answer sounds authoritative because the LLM is confident.
⚔️
Contradictory Facts
Multiple documents in the knowledge base say different things about the same topic. The retriever returns both, and the LLM either picks one arbitrarily, hedges confusingly, or blends them into a new "fact" that matches neither source.
Like asking 3 colleagues the same question and getting 3 different answers — then being asked to tell your manager "what the company policy is".
Real impact: A customer support bot is trained on a knowledge base where the product FAQ says "returns within 30 days" and a separate policy doc says "returns within 14 days". The bot sometimes says 30, sometimes 14 — destroying customer trust.
📄📄📄
Duplicate Content
The same document, or near-identical documents, are indexed multiple times. When the retriever fetches the "top 5 most relevant documents", 4 of them are copies of the same content — wasting the LLM's context window and crowding out genuinely different, useful sources.
Like going to a library where the same book is on 4 different shelves — so when a librarian grabs you 5 books on a topic, you get 4 copies of the same thing and only one truly different source.
Real impact: A legal research RAG system has a key case study document indexed 6 times under different filenames. The LLM overweights this single case and misses other relevant precedents, producing a one-sided legal summary.
🏷️
Poor Metadata & Indexing
Documents exist in the knowledge base but have bad titles, wrong tags, or no meaningful description. The retriever searches by keywords and semantic similarity — if a document is poorly labelled, it won't be found, even if it's the perfect answer to the user's question.
Like a library where every book spine says "Document" with a random number instead of the title and author. You know the answer is in there — but there's no way to find it.
Real impact: A product manual titled "Manual_FINAL_v3_USE_THIS_ONE.pdf" with no description or keywords. When a customer asks about installation steps, the retriever skips it entirely and the LLM makes up a plausible-sounding but wrong answer — a hallucination.

Core Responsibilities

  • Ensure DQ practices meet regulatory requirements (GDPR, BCBS 239, SOX, HIPAA)
  • Run the DQ maturity assessment program and publish the annual DQ maturity report

  • Publish and maintain the organization's Master Data Management (MDM) standards
  • Define and enforce data naming conventions, data type standards, and coding standards
  • Manage the Data Steward network — training, certification, and performance frameworks
  • Conduct quarterly cross-domain DQ reviews with all Data Owners
  • Produce the enterprise DQ scorecard and present findings to the executive team
  • Lead data quality incident post-mortems for P0/P1 events
What "Good" Looks Like: Every dataset in the organization has a published owner, steward, and quality SLA in the data catalog. The governance framework is understood by non-technical stakeholders. Regulatory audits consistently pass the DQ evidence review. New data initiatives include a governance checkpoint before going live.
⚠️ When this role is absent: Each domain invents its own DQ standards, metrics are defined inconsistently across teams, regulatory audits reveal undocumented data flows, and there is no mechanism to resolve disputes about whose version of "revenue" is correct.

Key DQ Activities

The DQ Issue Escalation Pathway

When a data quality issue is detected, it follows a structured escalation pathway. The path determines who is responsible for resolution at each stage and what happens if it isn't resolved within the defined SLA.

1
Automated Detection → DQ Analyst
A monitoring check fails and triggers an alert. The DQ Analyst receives the alert, triages severity (P0–P3), and begins investigation. They determine whether this is a pipeline issue, a source issue, or a business rule issue.
⏱ Target: Alert reviewed within 2 hours for Tier 1 tables
2
Technical Fix → Data Engineer
If the root cause is in the pipeline, the DQ Analyst creates a ticket assigned to the Data Engineer. The Engineer fixes the pipeline, deploys to production, and confirms the DQ check passes. For source system issues, the Engineer coordinates with the source system owner.
⏱ Target: P1 pipeline fix deployed within 24 hours
3
Business Rule Decision → Data Steward
If the issue requires a business judgment (is this value actually wrong, or has the business definition changed?), the DQ Analyst escalates to the Data Steward. The Steward either approves an exception with documented reasoning or confirms it is a genuine error requiring correction.
⏱ Target: Steward decision within 48 hours
4
Policy or Cross-Domain Issue → Governance Lead
If the issue exposes a gap in the DQ policy, requires a change to the enterprise standard, or involves a dispute between two domain teams, the Governance Lead is engaged. They adjudicate, update policy if needed, and publish updated guidance.
⏱ Target: Governance decision within 5 business days
5
Resource or Strategic Decision → Data Owner
If the fix requires significant engineering investment, a vendor change, or a strategic decision about acceptable risk, the Data Owner is the final escalation point. They approve or reject the remediation investment, set the risk acceptance policy, and communicate the decision to stakeholders.
⏱ Target: Owner decision within the next steering committee (max 2 weeks)

Step 1 — Talk to the People Who Use the Data

Good answer

"We look at a lot of dashboards." → Ask: "Which one would you miss most if it was wrong tomorrow morning?"

Why ask this: This maps decisions to specific database tables. The answer tells you exactly which tables to monitor in Tier 1.
Good answer
"The revenue dashboard pulls from fct_orders and dim_customers. If either of those is wrong or delayed, we can't trust anything else."

"Our CRM data." → Ask: "Which specific tables or reports? What does the data analyst on your team pull every morning?"

Why ask this: Past incidents reveal where checks are missing. If a team has experienced a specific problem, that exact scenario needs a check to prevent it recurring.
Good answer

Vague answer (dig deeper)
"Sometimes numbers look weird." → Ask: "Can you show me a specific example? What did you do when you noticed it?"

Why ask this: This question quantifies business impact and helps you prioritise. Data with a £1M/day cost if wrong goes in Tier 1; data with a £100 cost goes in Tier 4.

Step 3 — Write Down What "Good Data" Looks Like

For each important table, you need to define what correct data looks like across the 6 DQ dimensions. Think of this like writing a quality checklist for a delivery: "the package must arrive undamaged ✓, within 2 days ✓, with the correct address label ✓." For data, each dimension is one item on the checklist:

Completeness
Are all the important fields filled in?
Example: Every order must have a customer ID, an amount, and a date. An order without an amount is useless — you can't know how much was sold.
Rule: order_amount must NOT be NULL
Alert if: NULL rate > 0%
Accuracy
Are the values actually correct?
Example: An order amount of £-500 or £9,999,999 is suspicious. Real orders for this business range between £10 and £5,000.
Rule: order_amount between £10–£5,000
Alert if: any value outside this range
Validity
Is the format/value allowed?
Example: The status field should only contain "pending", "shipped", or "delivered". A value of "done" or "yes" is not a valid status — someone made a typo.
Rule: status IN ('pending','shipped','delivered')
Alert if: any other value found
Uniqueness
Are IDs and unique fields truly unique?
Example: Every order should have a unique order ID. If the same order ID appears twice, it means the order was loaded twice — so revenue will be double-counted.
Rule: order_id must be unique (no duplicates)
Alert if: any duplicate order_id found
Timeliness
Did the data arrive on time?
Example: The orders table should update every hour. If it hasn't updated in 4 hours, the data is stale — any dashboard showing "current" orders is actually showing old data.
Rule: max(loaded_at) must be within last 4 hours
Alert if: last load was >4 hours ago
Consistency
Does it match across systems?
Example: The total revenue in your data warehouse should match what the finance system shows. If they differ by more than 0.1%, something went wrong somewhere.
Rule: DW revenue = Finance revenue ±0.1%
Alert if: variance exceeds 0.1%

Step 4 — Connect Data Quality to Business KPIs

A KPI (Key Performance Indicator) is a number that tells you how well something is performing. Just as a business tracks "revenue" and "customer satisfaction", a DQ programme needs its own KPIs to measure whether data quality is getting better or worse over time. Here are the 5 most important ones:

📊 Overall DQ Trust Score
In plain English: A single number (0–100) that summarises how healthy your data is across all tables and dimensions. Like a credit score — but for your data. Executives can understand this instantly without knowing what a "null rate" is.
Example: "Our DQ Trust Score is 94.2 this week, up from 91.8 last month. Our target is 97."
Target: ≥ 95 for a mature DQ programme
✅ Check Pass Rate
In plain English: Out of all the automated checks that ran this week, what percentage passed? If 950 out of 1,000 checks passed, your pass rate is 95%. A declining pass rate is an early warning signal that data health is deteriorating.
Example: "This week 1,240 checks ran. 1,228 passed. Pass rate = 99.0% ✅"
Target: ≥ 98% — anything below 95% needs immediate investigation
💰 Revenue Accuracy Rate
In plain English: What percentage of your revenue figures match the finance system exactly? This is the check that finance teams care about most — even a 0.5% variance on £10M revenue means £50,000 is unaccounted for.
Example: "Today's DW revenue = £2,847,221. Finance system = £2,847,088. Variance = 0.005% ✅"
Target: ≥ 99.9% accuracy (within 0.1% tolerance)
⏱️ Mean Time to Resolve (MTTR)
In plain English: When a data quality issue is detected, how long does it take from the alert firing to the problem being fixed and confirmed? A lower MTTR means your team responds quickly and efficiently. High MTTR means issues linger and damage spreads.
Example: "This month's average MTTR for P1 issues: 2.4 hours. Last month: 5.1 hours. Improving ↓"
Target: P1 < 2 hours · P2 < 4 hours · P3 < next business day
🔄 Pipeline SLA Adherence
In plain English: Did your data pipelines finish on time? If the finance team needs last night's sales data by 7am, and the pipeline doesn't finish until 8:30am, you've missed the SLA. Tracking this shows how reliable your data delivery is.
Example: "48 pipelines ran overnight. 47 completed within their SLA window. 97.9% SLA adherence ✅"
Target: ≥ 99% SLA adherence for Tier 1 tables
📋 Customer Record Completeness
In plain English: What percentage of customer records have all the important fields filled in — name, email, phone, country? Incomplete customer records mean personalisation fails, campaigns go to the wrong people, and support teams can't contact customers properly.
Example: "Out of 284,000 customer records: 276,800 have all required fields. Completeness = 97.5% ✅"
Target: ≥ 98% completeness for marketing-active customers

🔧 What are "engineering requirements"?
Now that you know what to check (from Chapter 5), you need to understand where your data lives and how it moves before you can set up monitoring. Understanding Your Data Stack — Where Does Data Live?

Before monitoring data, you need to know its journey. Data travels through several "layers" from its origin (a CRM, a website, a finance system) to the dashboards and AI models that use it. DQ checks can be placed at any of these layers — and the earlier you catch a problem, the cheaper it is to fix.

DQ checks can be placed at any layer. The earlier a problem is caught in the pipeline, the cheaper and simpler it is to fix.

Step 1 — Take Inventory of Your Data Landscape

Before you can monitor your data, you need to answer four questions about your existing setup. This is your engineering survey — like a builder measuring rooms before laying flooring:

1
🗺️
Pipeline Inventory
Write down every data pipeline you have: where data comes from, which tool moves it, how often it runs, and which tables it loads. You cannot monitor pipelines you don't know exist.
Questions to answer: What source systems send us data? (CRM, ERP, website logs) → Which tool does the moving? (Fivetran, custom Python script) → How often? (hourly, daily, weekly)
2
🏗️
Infrastructure Assessment
Identify what data platform (cloud warehouse) you're using and what orchestration tool schedules your pipelines. This determines which DQ tools are compatible with your setup.
Questions to answer: Where is data stored? (Snowflake / BigQuery / Databricks) → What schedules pipelines? (Airflow / dbt Cloud / Prefect) → What runs transformations? (dbt / Spark / SQL)
3
🔐
Access & Permissions
Work out what access your DQ monitoring system needs. The golden rule: monitoring tools should be able to READ data to check it, but should never be able to WRITE or DELETE production data.
Example: Your DQ monitoring service account gets: read-only access to all production tables + write access only to the dq_monitoring.check_results table where it logs results. Nothing else.
4
📋
Existing Check Inventory
Before adding new checks, find out if any already exist. Many data teams have informal checks — SQL scripts someone runs manually, a dbt test file, a cron job. Catalogue these so you build on them rather than duplicate them.
Common places to look: dbt schema.yml files (look for "tests:" sections) → Airflow DAGs (look for validation tasks) → Slack/email history (find where people manually flag data issues)

Step 2 — Learn from Past Pipeline Problems

Before deploying new checks, look at your incident history. Every past data problem tells you exactly where a check was missing. Here are the most common pipeline issues — explained simply:

🐢
Late Data Arrivals
Weekly
What happens: Data was supposed to arrive at 6am, but the source system was slow. It arrives at 10am. By then, the morning dashboard shows yesterday's numbers and the sales team is making decisions on stale data.
Like a newspaper that's supposed to arrive at 7am but shows up at noon — it's technically correct, but the news is useless by then.
→ Fix with: Freshness check — alert if data hasn't arrived within the expected window
🔀
Schema Changes
Monthly
What happens: A developer on the CRM team renames a column or adds a new required field. Your pipeline doesn't know about the change and either breaks, loads NULLs, or silently loads the wrong data into the wrong column.
Like a form that always had 5 fields, but someone added a 6th field to the paper form — but your system still only reads 5, so the 6th is always blank.
→ Fix with: Schema drift detection — alert if column names, types, or counts change
📉
Volume Anomalies
Weekly
What happens: Yesterday the orders table had 4,200 rows. Today it has 12 rows — or 41,000 rows. Either something went wrong with the pipeline (too few rows = failed load) or data was loaded twice (too many rows = duplicate load).
Like a factory that normally produces 500 units a day. If it suddenly reports 10 units or 5,000 units — something went very wrong, even if you don't know exactly what yet.
→ Fix with: Row count check — alert if volume deviates >20% from the 7-day average
👯
Duplicate Records
Occasional
What happens: A pipeline runs twice by accident — maybe it was manually re-triggered after a failure, or retry logic kicked in before the first run finished. The same data lands in the table twice, doubling revenue, customer counts, and transaction totals.
Like accidentally photocopying every page of a report before submitting it — the total page count is double, but every insight is wrong.
→ Fix with: Uniqueness check on primary keys — alert if any ID appears more than once
🕳️
NULL Floods
Occasional
What happens: A bug in the source system or pipeline suddenly causes a critical field — like customer email or order amount — to be empty for thousands of rows. Everything looks normal (the pipeline "ran successfully") but the data is useless.
Like a printing error where every page of a report prints fine — except the "Total Revenue" column is blank on every row. The report looks complete but is meaningless.
→ Fix with: Null rate check — alert if null rate on critical columns exceeds 1%
🔡
Format / Type Changes
Occasional
What happens: A date field that always contained "2025-03-06" suddenly starts arriving as "06/03/2025" or "March 6, 2025". Downstream SQL queries break, dashboards show errors, and time-series analysis becomes unusable.
Like a spreadsheet where the date column worked fine for months, then someone changed the cell format — now Excel reads every date as a number and all your charts break.
→ Fix with: Format validation — check dates match expected pattern, numbers are numeric

?

?

?

?

?

Common QA Challenges in Data Platforms

⚠️ Know the pitfalls before you build

Every data platform runs into the same category of quality problems — regardless of which tools or cloud provider you use. Understanding these challenges now means you can design your checks, alerts, and ownership model around them from day one, rather than discovering them after they've already broken something important.

💡 Note: This section covers platform-level design challenges — things to plan around before you build. For runtime pipeline issues (schema drift, duplicate ingestion, type errors, stale loads) and their step-by-step fixes, see Chapter 13 · Fixing Data Pipeline Issues.
🔇 Silent Pipeline Failures High Impact

A pipeline job reports "success" in the scheduler but the output is empty, partial, or incorrect. No error was thrown — the job simply ran on no data, or a filter was mis-applied, or a JOIN returned zero rows. Downstream consumers see a green status light but receive wrong data.

Like a printer that shows "Print complete" but the paper tray was empty — the job finished, but nothing actually came out.
⚠ Output table suddenly has 0 rows ⚠ Row counts drop by >20% overnight ⚠ All values in a column are NULL
Never trust exit codes alone. Add post-run row-count assertions, NULL-rate checks, and min/max value range checks. A job that produces an empty table should trigger an alert, not a green tick.
🔬 Test Coverage Gaps Medium Impact

Teams write DQ checks for the columns they think about — but not for every column that matters. A field like "discount_percentage" or "refund_flag" might have no check at all. An edge case (a NULL where there should never be a NULL) goes undetected for months because no one defined a rule for it.

Like a food safety inspector who only checks the ingredients list but never the expiry date or storage temperature — the parts they didn't inspect are where the problem hides.
⚠ Issues in fields with no existing checks ⚠ Metrics wrong despite all checks passing ⚠ New columns added without review
Run a coverage audit — list every column in your critical tables and confirm at least one DQ check exists for each. Treat unchecked columns as technical debt. Add checks every time a new field is introduced.
🏗️ Dev / Prod Environment Parity Medium Impact

Development and test environments use synthetic or sampled data that doesn't reflect the messiness of real production data. Checks pass in testing but fail in production because the real data has edge cases, unusual values, or volume patterns that the test data didn't include.

Like rehearsing a play on an empty stage and then performing for the first time in a full theatre — the extra complexity exposes problems that never showed up in rehearsal.
⚠ Checks pass in dev, fail in prod ⚠ Performance degrades at scale ⚠ Edge-case values absent from test data
Use anonymised or masked samples of real production data in your test environment. Run DQ checks against production data in read-only mode before deploying changes to pipelines.
🧩 Referential Integrity Breaks Very Common

A record in one table references an ID in another table that no longer exists — an orphaned foreign key. This creates silent data loss in JOINs: the orphaned rows simply disappear from queries without any error. Common after migrations, deletions, or when source systems don't enforce their own foreign key constraints.

Like a library catalogue that lists a book as "available on shelf 4B" — but when you go there, the book is gone. The record exists; the thing it points to doesn't.
⚠ JOIN results have fewer rows than expected ⚠ Metrics drop after a migration ⚠ IDs present in one table, absent in another
Run referential integrity checks as a standard post-ingestion test — count orphaned foreign keys for every critical relationship. Any orphan count above zero should generate an alert.
📉 Metric Drift Without a Code Change Very Common

A KPI shifts slowly over weeks — conversion rate creeps down, average order value drifts up — but no pipeline or business logic changed. The cause is the underlying data changing in character: new customer segments, seasonal patterns, or a source system that started sending different values for the same field.

Like a set of scales that slowly becomes miscalibrated — each individual reading looks plausible, but over time everything is consistently off in a direction nobody noticed.
⚠ KPI trends diverge from expectations ⚠ Distribution of a field shifts over weeks ⚠ New value categories appear in data
Monitor statistical distributions, not just counts. Set up anomaly detection on key metric trends so that gradual drift — not just sudden spikes — triggers an investigation before stakeholders notice.

Connecting & Deploying
DQ Checks

Create the DQ Environment — Your Dedicated Control Room

Before deploying any checks, set up a dedicated space for your DQ monitoring system — separate from production data. Think of it like a security control room: it can see everything in the building (read access to all data), but it operates in its own space and cannot modify what it monitors.

🏗️ What "dedicated DQ environment" means in practice

Create a separate schema in your data warehouse (e.g. dq_monitoring) that stores only DQ check results — never the actual business data. Create a service account with read-only access to all production tables, and write access only to this dedicated schema. This prevents any accidental changes to real data.

Import Metadata — Take a Snapshot of Your Data Landscape

Before checks run, you need a baseline — a record of what your data normally looks like. This metadata snapshot (row counts, null rates, min/max values, last load times) becomes the reference point against which future checks are compared. Without a baseline, you can't tell if today's data is "normal" or suspicious.

💡 What gets stored in the metadata table?

Your DQ environment records the results of every check — not the actual business data. For each check it stores: which table was checked, what the check found (e.g. "2.3% null rate"), whether that passed or failed, and when it ran. Over time this creates a history so you can spot trends — like a column that's been slowly getting worse for weeks.

Deploy Your Data Quality Checks — Step by Step

Think of deploying a DQ check like installing a smoke detector. You don't just stick it to the ceiling — you test it works, decide how loud the alarm should be, make sure someone will hear it when it goes off, and set a reminder to check the battery every month. Each step below is essential.

The 5-Step Deployment Journey
1
✏️
Write the Check
This is where you define exactly what "good data" looks like. You write a rule — in SQL, YAML, or a tool like Great Expectations — that can be run automatically to test your data.
What actually happens: A SQL query counts how many rows have a null value in the customer_email column. If that number exceeds 1% of total rows, the check fails.
Example: In dbt, you add two lines to a YAML config file: - not_null and - unique under customer_id. That's it — dbt will enforce both rules every time the model runs.
💡 Beginner tip: Start with the most critical column in your most critical table. Write one check. Get it working end-to-end before writing 50 more.
2
🧪
Test in Staging — Never Go Straight to Production
A staging environment is a safe copy of your real data setup — changes here don't affect anything real. Always test your checks here first to make sure the rule is correct and the threshold is sensible.
What actually happens: You run the check against last week's real data (safely copied to staging). It might fire an alert for something that's actually fine — telling you the threshold is set too tight and needs adjusting.
Example: You set a threshold of "alert if null rate > 0.1%". But in staging you discover the marketing email field is historically 5% null (people opt out). You adjust the threshold to 8% — much more useful.
💡 Beginner tip: Look at 30–60 days of historical data in staging to understand "normal" before committing to a threshold. Guessing leads to noise.
3
📋
Log Results — Create Your Quality History
Every time a check runs, the result (pass or fail, score, timestamp) gets written to your DQ metadata table. This builds a history — so you can see trends over time, not just snapshots.
What actually happens: After the null check runs, a row is inserted: "Table: orders, Column: customer_email, Check: null_rate, Value: 0.8%, Threshold: 8%, Passed: true, Timestamp: 2025-03-06 06:02". Next week you can look back and see if the null rate has been creeping up.
Example: Without logging, you only know "today's check passed". With logging, you can see the null rate went from 0.5% → 0.8% → 1.2% over three weeks — catching a slow data leak before it becomes a crisis.
💡 Beginner tip: Never skip logging. A check that passes silently is worthless for trend analysis. The history is where the real value lives.
4
🔔
Configure Alerts — Tell the Right People When Something Goes Wrong
When a check fails, someone needs to know immediately — not when a user complains about bad data in a dashboard. Alerts route failures to the right person via Slack, email, or PagerDuty based on severity.
What actually happens: You configure: "If this check fails AND it's tagged Critical, post to #data-incidents on Slack AND page the on-call engineer. If it's Warning, just post to #data-quality channel." Different problems reach different people.
Example: A null rate check on the revenue column is Critical — it pages an engineer at 3am if needed. A cosmetic issue on a marketing field is Low — it creates a ticket in Jira and no one is woken up.
💡 Beginner tip: Misconfigured alerts are the #1 cause of "alert fatigue" — where people start ignoring notifications because there are too many. Start with fewer, higher-quality alerts.
5
⏰
Schedule Execution — Make It Automatic
Checks that someone has to remember to run will be forgotten. Add your checks to the pipeline scheduler (Airflow, dbt Cloud, cron) so they run automatically after every data load — no manual work required.
What actually happens: In Airflow you add a "run_dq_checks" task at the end of your pipeline DAG. Every night after the data loads, DQ checks run automatically, results are logged, and any failures trigger alerts — all while you sleep.
Example: Your orders pipeline runs at 2am. DQ checks are scheduled to run at 2:30am. By 2:35am, results are logged and the team's Slack has a green ✅ status summary. If something failed, the alert was already sent at 2:31am.
💡 Beginner tip: DQ checks should always run after the data load, not before. You're validating the data that just arrived, not the data from yesterday.

Setting Alerting Thresholds — Not Too Tight, Not Too Loose

A threshold is the line between "this is fine" and "sound the alarm". Set it too tight, and your team will be bombarded with false alarms and start ignoring everything (called alert fatigue). Set it too loose, and real problems slip through undetected. There are three approaches:

Type 1
Static Threshold
A hard line that never changes. Used for rules that are always true — no exceptions, no nuance.
When to use it: Zero-tolerance situations.

Examples:
• Primary key must never be null → threshold: 0%
• Order ID must be unique → threshold: 0 duplicates
• Transaction amount must be positive → threshold: 0 negatives
Type 2
Relative Threshold
Based on how much something changes compared to recent history. Used when the absolute value isn't the problem — the sudden change is.
When to use it: Volume and count checks.

Examples:
• Alert if today's row count is >20% different from the 7-day average
• Alert if null rate jumped by more than 5 percentage points vs. yesterday
• Alert if revenue is >30% lower than same weekday last week
Type 3
Dynamic Threshold
Uses statistical modelling to define what's "normal" — accounting for weekly patterns, seasonal spikes, and growth trends. The threshold adjusts automatically over time.
When to use it: Metrics that grow or have seasonal patterns.

Examples:
• Order volume is always higher on Fridays — dynamic threshold accounts for this automatically
• Revenue grows 10% month-on-month — a static threshold from January would be wrong by December
🌱 Which threshold type should a beginner start with?

Start with Static for critical columns (nulls, uniqueness, foreign keys). Then add Relative for row counts and volume checks. Only graduate to Dynamic once you have at least 60 days of logged check history — you need the history to model "normal".

Custom Checks — When Standard Rules Aren't Enough

Standard checks (not null, unique, range) cover a lot — but every business has rules that only make sense in context. Custom checks capture business logic that no off-the-shelf tool could guess. Here are four common types:

💰
Revenue Reconciliation Check
Compares the total daily revenue in your data warehouse against the number in your finance system. Even a small discrepancy can indicate a pipeline bug or missing data.
Rule: Sum of order_total in warehouse must match reported_revenue in finance system within 0.1% tolerance.
🔗
Cross-Table Referential Integrity
Checks that every record in one table has a valid matching record in another — even when they live in different schemas or databases.
Rule: Every customer_id in the orders table must exist in the crm.customers table. Orphaned orders indicate a sync failure.
⏱️
SLA Compliance Check
Verifies that data arrived within the contractual time window. If a partner sends you data every hour, a check confirms you actually got it — not just that something arrived.
Rule: The max(load_timestamp) for each feed must be within the last 65 minutes. If older than that, the feed has missed its SLA window.
📐
Business Logic Validation
Checks domain-specific rules that only make sense if you understand the business. These are the checks only your team can write — no tool can infer them automatically.
Rule: Subscription end date must always be after start date. Discount percentage must be between 0–100%. Refund amount must never exceed original order amount.

The Data Quality
Monitoring Process

🔍 What is Data Quality Monitoring? (Start Here)
Imagine you have a water pipe running through your building. You could wait until water starts dripping through the ceiling to know there's a problem — or you could install sensors that alert you the moment pressure drops abnormally, before any damage happens.

Data Quality Monitoring is those sensors for your data. Instead of waiting for a business analyst to notice a suspicious number in a report, or a customer to complain about a wrong invoice, monitoring systems automatically check your data at every step of the pipeline and alert the right people the moment something goes wrong.

The goal is simple: catch data problems before they reach the people who make decisions. Every hour earlier you catch an issue, you prevent hours of investigation, re-work, and lost business trust.

Without Monitoring vs With Monitoring

🚒 Without Monitoring — Fighting Fires
  • ✗ A data pipeline silently breaks at 2am. No one knows.
  • ✗ At 9am, a sales director opens their dashboard and sees revenue drop 80%.
  • ✗ They escalate to the CEO. Panic meeting called.
  • ✗ A data engineer spends 4 hours tracing the issue manually.
  • ✗ Root cause: a missing date column in an upstream table. A 5-minute fix — after 7 hours of chaos.
  • ✗ Trust in data is damaged. Leadership questions every report for weeks.
✅ With Monitoring — Early Warning System
  • ✓ A pipeline runs at 2am. An automated check detects a NULL date column.
  • ✓ At 2:04am, a Slack alert fires to the on-call data engineer.
  • ✓ The downstream dashboard is automatically flagged as "stale — do not use".
  • ✓ The engineer fixes the pipeline at 7am before business hours.
  • ✓ The dashboard refreshes with correct data at 8:45am.
  • ✓ The sales director opens their dashboard at 9am and sees accurate numbers. Nothing to report.

The 6-Step DQ Monitoring Cycle

Data quality monitoring is not a one-time task — it is a continuous loop. Every time a problem is found and fixed, it feeds learning back into better checks and thresholds. This cycle runs every day, automatically.

Figure 8.1 — The DQ Monitoring Cycle runs continuously. Each completed loop makes the next cycle faster and smarter.

1
🔔
Detect — Your Automated Watchman
In plain English: Automated checks run silently in the background every time your data pipelines load new data — like a security system that checks every door and window automatically, without anyone having to remember to do it.
What actually happens: After each pipeline run (e.g., overnight ETL, hourly API sync), a suite of pre-defined checks execute against the new data. These checks test things like: are there any NULL values in critical fields? Has the row count dropped unexpectedly? Are all values within expected ranges?
Example: Every morning at 6am, after the sales pipeline refreshes, 240 automated checks run across 18 tables. One check finds that the orders table has 0 rows today (yesterday it had 4,200). This is flagged immediately — before any analyst opens a report.
💡 Beginner tip: Think of this like a spell-checker that runs on your data automatically. You don't have to manually read every sentence — it highlights problems for you to review.
2
🔎
Triage — Is This Actually a Problem?
In plain English: Not every alert is an emergency. Triage is the process of quickly classifying how serious an issue is — like a hospital A&E nurse who sees 30 patients at once and decides who needs immediate care versus who can wait.
What actually happens: A DQ analyst (or an automated system) looks at the failing check and asks: How many rows are affected? Which downstream reports or dashboards depend on this data? Is this a real data problem or a known acceptable variation? How urgent is the business impact?
Example: The 0-row orders table alert arrives. The analyst checks: Is today a bank holiday? No. Did the source system go offline? Checking... yes — the CRM API timed out at 5:52am. This is a P1 Critical issue. Downstream: revenue dashboard, CEO weekly report, finance reconciliation. Escalate immediately.
💡 Beginner tip: Triage is about making a fast decision: "Do I need to wake someone up right now, or can this wait until tomorrow's standup?" Getting this wrong in either direction causes problems — too many urgent alerts causes fatigue; too few means problems are missed.
3
🧩
Investigate — Finding the Root Cause
In plain English: Once you know there's a problem and how serious it is, you need to find out why it happened. This is detective work — tracing the issue back to its origin, not just treating the symptom.
What actually happens: The engineer looks at data lineage (which systems feed this table), compares today's data to yesterday's baseline, checks pipeline logs for errors, looks at sample bad rows to spot a pattern, and asks upstream teams if anything changed.
Example: Investigating the 0-row orders issue: The data engineer checks pipeline logs → finds a "Connection timeout" error at 5:52am → checks the CRM API status page → the API had a 2-hour outage → the pipeline retry logic was set to 1 retry (should be 3) → root cause identified: insufficient retry configuration, not a data issue. Fix is in the pipeline config, not the data itself.
💡 Beginner tip: Always ask "why did this happen?" five times (the "5 Whys" technique). Stopping at the first answer usually means you're treating a symptom. The zero rows weren't the root cause — the API timeout wasn't the root cause — the misconfigured retry was the root cause.
4
📣
Escalate — Getting the Right People Involved
In plain English: Some issues can be fixed by the data team alone. Others need the source system team, the business owner, or even executives to be informed. Escalation means routing the problem to whoever has the authority and knowledge to fix it — with the right level of urgency.
What actually happens: Based on the triage severity and root cause, the issue is assigned to the right resolver with a clear description: what failed, what the impact is, what the root cause appears to be, and what action is needed. A P1 issue goes to the on-call engineer and the business owner immediately. A P3 issue goes into the team backlog.
Example: The orders pipeline issue is escalated to: (1) the CRM vendor's support team to report the API outage, (2) the data engineering team to fix the retry configuration, (3) the Head of Sales via Slack to inform them the revenue dashboard will be delayed by 2 hours, (4) the finance team to pause the morning reconciliation run.
💡 Beginner tip: Good escalation means communicating three things: "Here's what broke, here's who is affected, here's what we're doing about it." Vague escalations ("data is wrong") cause panic. Clear escalations ("orders table has 0 rows due to a CRM API outage, estimated fix by 9am") allow people to plan.
5
🔧
Resolve — Fix It, Prove It's Fixed
In plain English: Apply the fix, then verify the fix worked by re-running the checks that originally failed. Never close an issue by assumption — always close it with evidence. Log what was done so the next person doesn't have to investigate the same issue from scratch.
What actually happens: The engineer applies the fix (code change, data correction, pipeline re-run). The failed DQ checks are re-executed. If they pass, the incident is closed with a written resolution note. If they fail again, investigation continues. Downstream systems are un-flagged and business users are notified that data is correct.
Example: The retry config is updated from 1 to 3 retries. The pipeline is manually re-triggered and runs successfully. The orders table now shows 4,187 rows (slightly fewer than yesterday — normal variation). All 240 checks pass. The revenue dashboard updates at 9:15am. The incident is closed with note: "CRM API outage caused pipeline failure. Retry config updated to prevent recurrence."
💡 Beginner tip: "I think it's fixed" is not the same as "the checks are passing again." Always re-run the checks. This protects you, documents the resolution, and builds the evidence trail that shows your monitoring is working.
6
🛡️
Prevent — Learn So It Doesn't Happen Again
In plain English: Every incident is a learning opportunity. After resolution, the team asks: "What can we put in place so this either doesn't happen again, or is caught faster next time?" This is what separates a reactive team from a proactive one.
What actually happens: A short post-mortem is written for all P1/P2 incidents. The team identifies: Was there a check we should have had but didn't? Should the alert threshold be lower? Was the escalation path unclear? Should retry logic be standardised? New checks or process improvements are added to the backlog.
Example: Post-mortem action items from the orders incident: (1) Add a check that fires if orders table row count is 0 at any time (not just after pipeline run). (2) Standardise retry configuration across all CRM-dependent pipelines. (3) Add a "dashboard status" flag that auto-shows a warning banner when data is stale. Three improvements from one incident.
💡 Beginner tip: High-maturity data teams treat every incident as a gift — it tells them exactly where their monitoring had a blind spot. Teams that skip this step fix the same issues over and over. Teams that do this step get faster and smarter with every cycle.

Common DQ Checks — What They Catch & Why They Matter

Here are examples of the most common automated checks used in practice, with plain-English explanations of what they catch:

🕳️
NULL Check
Completeness
Checks that important fields are not empty. A customer record with no email address, or an order with no amount, is incomplete and will break downstream processes.
✓ PASS: customer_email = "jane@example.com"
✗ FAIL: customer_email = NULL (1,240 rows)
→ Alert: 3.2% null rate exceeds 1% threshold
👥
Uniqueness Check
Uniqueness
Checks that values that should be unique (like customer IDs or order numbers) don't appear more than once. Duplicates cause double-counting in reports and inflate metrics.
✓ PASS: order_id = "ORD-4421" appears once
✗ FAIL: order_id "ORD-4421" appears 3 times
→ Alert: 847 duplicate order_ids found
📏
Range / Boundary Check
Validity
Checks that numeric values fall within expected bounds. A discount of 150%, a customer age of 312, or a negative order quantity are all technically valid numbers but clearly wrong.
✓ PASS: discount_pct = 15 (within 0–100)
✗ FAIL: discount_pct = 150 (exceeds 100%)
→ Alert: 23 rows with discount > 100%
⏱️
Freshness Check
Timeliness
Checks that data is up to date. A sales dashboard showing yesterday's numbers as "today" because the pipeline failed silently — without a freshness check, no one would know.
✓ PASS: max(loaded_at) = today 06:02 UTC
✗ FAIL: max(loaded_at) = yesterday 06:05 UTC
→ Alert: Table is 24h stale, SLA = 6h
📊
Row Count / Volume Check
Completeness
Checks that the table received approximately the expected number of rows. A sudden drop (pipeline failure) or spike (duplicate load) in row count is a strong signal something went wrong.
✓ PASS: row_count = 4,187 (baseline ~4,200)
✗ FAIL: row_count = 0 (expected ~4,200)
→ Alert: 100% drop from yesterday baseline
🔗
Referential Integrity Check
Consistency
Checks that values in one table actually exist in the related table. An order for a product that doesn't exist in the products table, or a transaction for a customer that doesn't exist in the customers table, is an orphaned record.
✓ PASS: order.product_id exists in products
✗ FAIL: 312 orders link to deleted product IDs
→ Alert: 312 orphaned records found

Alerting, Thresholds
& Anomaly Detection

🚨 Think of alerts like smoke detectors in your building
Imagine a building with no smoke detectors. The only way you'd know there was a fire is when you smell smoke — by which point the damage is already serious. Alerts in data quality work the same way.

Without alerts, you only discover data problems when someone complains — a wrong number in a report, a broken dashboard, a confused customer. By then, the bad data has already spread everywhere. With a well-tuned alert system, you're notified the moment something goes wrong — before anyone downstream even notices.

The goal isn't to have the most alerts — it's to have the right alerts: loud enough to be noticed, specific enough to be actionable, and rare enough that people actually take them seriously.

Without Alerts vs With Alerts — The Difference in Practice

❌ Without a Good Alert System
  • •A pipeline silently drops 30% of orders overnight. Nobody knows.
  • •Finance publishes an incorrect revenue report at 9am. It reaches the CEO before anyone catches it.
  • •A customer emails support: "Why is my order history wrong?" — that's your first signal of a 3-day-old data problem.
  • •Engineers spend hours investigating what went wrong and when — because there's no log of what changed.
  • •Trust in data erodes. People start "double-checking" everything manually — wasting hours every week.
✅ With a Well-Tuned Alert System
  • •At 2:31am, Slack: "P1 Alert — orders table row count dropped 31%. On-call engineer paged."
  • •Engineer fixes the pipeline at 3am. Finance publishes the correct report at 9am as planned.
  • •Zero customers are affected. Zero incorrect reports published. No one outside the data team even knew.
  • •The investigation takes 40 minutes — not 4 hours — because the alert included exactly which check failed and on which table.
  • •The team builds confidence in the system. Data becomes a trusted asset, not a source of anxiety.

⚠️ The Alert Fatigue Problem — More Alerts Is Not Better
Alert fatigue happens when engineers receive so many notifications that they stop reading them carefully — or start dismissing them reflexively. This is one of the most dangerous states a DQ programme can fall into, because when a real P1 fires, it gets ignored in the noise.

Signs your alert system has fatigue: Engineers mute the DQ Slack channel. P1 alerts go unacknowledged for hours. The same alert fires every single day. People say "oh that always fires" and ignore it.

Fix it by: Reviewing which alerts fire most frequently and either fixing the underlying issue or adjusting the threshold. A good rule of thumb: if a check fails more than once per week on average, either the check rule is wrong or the underlying data issue needs a permanent fix — not just repeated alerts.

Statistical Anomaly Detection — When You Can't Predict "Normal"

📈 What is Anomaly Detection?
Imagine you track how many orders come in every day for a year. Monday–Friday you get about 1,000 orders. Weekends are quieter — around 400. Black Friday you get 8,000.

A static threshold (e.g. "alert if orders < 500") would fire every weekend even though that's completely normal. And it would miss a weekday where orders dropped from 1,000 to 600 — which might indicate a real problem.

Anomaly detection is smarter. Instead of a fixed number, it looks at the recent history and asks: "Is today's value unusual compared to what we'd normally expect for a day like this?" It learns your patterns — including weekday/weekend cycles, seasonal trends, and growth curves — and flags only genuinely unexpected deviations.

How Does Anomaly Detection Decide What's "Normal"?

The most common method is called the Z-score. Here's all you need to know without any maths: the system looks at the last 30 days of data, calculates what's typical, and measures how far today's value is from that typical value — in units called "standard deviations". The further away, the more unusual.

In everyday terms: If the last 30 days of order counts averaged 1,000 per day and today you only got 400, that's a Z-score of about −3. That's so far outside the normal range that the system knows something is genuinely wrong — not just a quiet day. It fires a P1 alert. Meanwhile if orders are 950 (Z = −0.5), the system does nothing because that's completely normal variation.

4 Common Types of Data Anomalies to Detect

📉
Volume Drop
Suddenly far fewer rows than expected arrive in a table. Usually means a pipeline failed partway through, or a source system stopped sending data.
Example: Orders table normally has ~1,000 rows added daily. Today only 12 arrived. The ETL job failed after the first batch.
📈
Volume Spike
Suddenly far more rows than expected — could indicate duplicate data loading, a data backfill being accidentally re-run, or a genuine business event.
Example: User events table gets 3 million rows today vs a normal 200,000. Investigation reveals an Airflow DAG re-ran 15 days of history by mistake.
📊
Distribution Shift
The range or pattern of values in a column changes unexpectedly. The data is present, but the numbers look different from what's historically been normal.
Example: Product prices historically ranged £10–£500. Today a batch of products has prices between £1–£5. Likely a currency conversion bug — prices came in as USD cents, not GBP.
⏰
Freshness Anomaly
Data stopped arriving on schedule. The most recent record in the table is older than expected — meaning the feed is stale or the pipeline is stuck.
Example: The inventory table is normally updated every hour. The last update was 7 hours ago. The warehouse system's API credentials expired and the feed silently stopped.

4 Rules for Healthy Alerting — Beginner Best Practices

Rule 1 Start with fewer, higher-quality alerts
It's tempting to alert on everything. Resist. Start with only P1 and P2 for your most critical tables. Add more checks gradually once you've tuned the first ones.
Target: zero false positives on P1 alerts. Every P1 should be a genuine emergency that warrants waking someone up.
Rule 2 Route alerts to the right person
A P1 Revenue alert going to the wrong Slack channel is almost as bad as no alert. Map every alert type to the right team and individual before going live.
Example: Finance data issues → #finance-data-alerts. Marketing data → #marketing-data-health. Infra failures → on-call PagerDuty rotation.
Rule 3 Include context in every alert
An alert that says "DQ check failed" is useless. An alert that says "NULL rate on orders.customer_email jumped from 0.5% to 18% (threshold: 2%) — 3,400 rows affected since 02:15 UTC" is actionable.
Every alert message should answer: What failed? On which table/column? How bad is it? When did it start? How many rows are affected?
Rule 4 Review and tune monthly
Alerts that made sense in January may be wrong by July — your data volumes grow, patterns shift, and business rules change. Schedule a monthly 30-minute review of alert performance.
Ask monthly: Which alerts fired most? Were they genuine? Did any real issues get missed? Adjust thresholds and routing based on what you find.

Fixing
Source Data Issues

🍽️ Think of it like complaining to a restaurant vs. fixing it yourself
Imagine you ordered a meal and it arrived cold — but the problem wasn't in your kitchen, it was in the restaurant's. You can't just fix it yourself. You have to communicate the problem to the restaurant, give them evidence of what went wrong, and follow up until they fix it.

Source data issues work exactly the same way. The bad data is coming from a system your team doesn't control — a CRM, an ERP, a third-party API. You can detect the problem. You can escalate it. But you can't fix it alone. This chapter shows you how to investigate, confirm, communicate, and track resolution with the teams that own the source systems.

First: Make Sure It's Actually a Source Problem

Before you contact another team and say "your data is broken", you need to be certain the problem isn't in your own pipeline. Nothing damages team relationships faster than escalating an issue that turns out to be your own transformation bug.

🔴 Signs it IS a source problem
• The raw data (before any transformation) already shows the issue
• The same problem appears across multiple pipelines using the same source
• The source team recently made a change to their system
• The data looks wrong in the source system itself (CRM, ERP)
• The issue has been present since a specific date that matches a source deployment
🟢 Signs it's YOUR pipeline (not the source)
• The raw/staging data looks fine, but the output table is wrong
• Only one pipeline is affected, not others using the same source
• Your team recently deployed a code change
• The source data is correct in the source system
• The issue started exactly when a pipeline job ran or was modified

How to Investigate a Source Issue — 5 Steps

Before escalating anything, build a solid evidence pack. You want to walk into the conversation with the source team armed with facts, not assumptions:

1
Confirm it's reproducible — not a one-off
Run the check a second time, on a different day's data. A genuine source problem will show up consistently. A one-off network blip will not. Only escalate if it's reproducible.
Example: Email null rate was 18% today. Check yesterday's data — if it was 2%, this might be a one-day issue, not a systemic problem.
2
Check the raw data before your pipeline touched it
Look at your raw/staging layer — the data exactly as it arrived from the source, before any transformation. If the problem is already there, it's a source issue. If it appeared during transformation, it's yours to fix.
Example: Query your raw S3 or staging table — if emails are already missing there, the source system isn't sending them.
3
Gather specific examples with context
Don't say "emails are missing". Say "These 3,421 customer records loaded on 2025-03-06 have no email address. Here are 5 specific customer IDs." Concrete examples are 10× easier to investigate than vague descriptions.
Save a sample: customer_id, created_at, what's missing, and what it should contain based on business rules.
4
Quantify the business impact
How many rows are affected? Which reports are showing wrong numbers because of this? Is revenue being miscalculated? Is a compliance report incorrect? This determines urgency and gets the right level of attention from the source team.
Example: "3,421 customers can't receive order confirmation emails. We estimate £12,000 in orders may have poor post-purchase experience."
5
Find out when it started
Check your check history — when did this check last pass? This tells you how long the data has been bad and whether historical data needs to be corrected too. It also helps the source team identify which system change caused it.
Example: "The check last passed on March 1st. The null rate started rising on March 2nd. Did anything change in the CRM on that date?"

The 5-Why Technique — Find the Real Cause, Not Just the Symptom

When you find a data quality problem, it's tempting to fix the first obvious cause and move on. But that usually means the same problem comes back in a different form. The 5-Why technique makes you ask "why?" repeatedly until you reach the real root cause — the systemic gap that allowed the problem to happen in the first place.

Here's a real example — following the trail from a symptom all the way to the root:

❌ The Problem
Customer email completeness dropped from 97% to 71%. Thousands of customer records arrived with no email address. Marketing campaigns can't send. Support can't reach customers.
Why 1
The CRM integration pipeline loaded new customer records without email addresses. But why did the pipeline load records that were incomplete?
Why 2
The CRM export query was changed — it no longer included a JOIN to the contacts table where email addresses are stored. But why was the query changed?
Why 3
A CRM system admin updated the export query template to improve performance — without knowing that the data team's pipeline depended on it. But why didn't the data team know about the change?
Why 4
There was no change notification process between the CRM team and the data team. Nobody knew they needed to tell each other about configuration changes. But why was there no process?
✅ Root Cause — Stop Here
No data contract existed between the CRM team and the data team. Nobody had formally agreed what the CRM would provide, in what format, and who would be notified before changes. The fix isn't "reconnect the JOIN" — it's creating a data contract and change notification process so this can never happen silently again.
📩 How to Communicate a Source Issue to Another Team

When you reach out to a source team, be specific, non-blaming, and make it easy for them to help. Here's a template that works:

The Exception Register — When You Can't Fix It Right Now

Sometimes a source data issue genuinely can't be fixed quickly — it might require a major system change, third-party vendor involvement, or a budget decision. In these cases, you don't just leave it open forever. You formally accept the risk by logging it in a DQ Exception Register with leadership sign-off.

Think of it like a known pothole in the road that the council hasn't fixed yet. You mark it on the map, put up a warning sign, and make sure everyone knows it's there — so nobody drives into it by accident.

Fixing Data
Pipeline Issues

🏭 Think of it like a factory assembly line
Imagine a factory where raw materials go in at one end and finished products come out at the other — passing through several machines in sequence. If one machine in the middle breaks, everything downstream is affected. Products come out wrong, or not at all.

A data pipeline works the same way. Raw data enters at one end (source systems), passes through transformations (cleaning, joining, aggregating), and arrives as a ready-to-use table at the other end. When something goes wrong in the middle, the finished output is wrong — and it's your team's job to find which step broke and fix it.

The good news: unlike source issues (Chapter 12), pipeline problems are entirely within your control. You own the code. You can debug it, fix it, and deploy the solution yourself.

How to Investigate a Pipeline Issue — 4 Steps

When a DQ check flags a failure, don't start randomly changing code. Follow this systematic process — it usually finds the cause in under 20 minutes:

1
📋
Read the orchestrator logs first
Open Airflow, Prefect, or dbt Cloud and find the failed run. Look at the error message, which task failed, and when. This is your starting point — don't guess, read the logs.
Look for: "KeyError: 'customer_email'" or "Division by zero in revenue_calc" — these tell you exactly which step broke and why.
2
🔬
Check the raw/staging data — did the extract succeed?
Before assuming the transformation broke, check if the source data arrived correctly. Query your raw or staging layer. If the raw data is already wrong, it's a source issue (Chapter 12), not a pipeline issue.
SELECT COUNT(*) FROM raw.orders WHERE load_date = today — if this returns 0, the extract failed, not the transformation.
3
🔧
Reproduce the issue in isolation
Copy a small sample of the failing data and run just the broken transformation step manually — in a notebook, SQL editor, or dbt. Reproduce the error, fix it, then confirm it passes on that sample before deploying to production.
Run the dbt model with a limit: dbt run --select my_broken_model against a small test dataset — much faster than a full run.
4
🕵️
Compare to the last successful run — what changed?
Find when the pipeline last ran successfully. Then ask: what changed between that run and the failing one? New source data? A code deployment? A schema change? An infrastructure event?
Check git history for recent commits, your data catalog for schema changes, and cloud dashboards for infrastructure events around the failure time.

The 5 Most Common Pipeline Issues

Most pipeline DQ problems fall into five categories. Here's what each looks like, why it happens, and how to fix it:

🔄
Schema Drift — "A Column Disappeared (or Appeared)"
What you see: A column that should be in the output is suddenly missing. Or a new column arrived in source data but isn't loading. Downstream reports break with "column not found" errors.
🧩 Like a supplier changing a product's packaging without telling you — your sorting machine expected a standard size, and now everything jams.
✅ Fix: Update your transformation to handle the changed column. Add schema drift detection checks so you're alerted the moment a source schema changes — before it breaks anything downstream.
👯
Idempotency Failure — "Running It Twice Created Duplicates"
What you see: After a pipeline re-run (common after a failure), you suddenly have duplicate records — the same order appears twice, the same customer exists twice. Uniqueness checks start failing.
🖨️ Like a photocopier that gets stuck and makes two copies of every page — everything looks right until you count and find everything is doubled.
✅ Fix: Change your pipeline from INSERT (adds rows every time) to MERGE (updates existing rows and only adds genuinely new ones). A MERGE is safe to run multiple times — it gives the same result each time, which is what "idempotent" means.
🔡
Type Casting Error — "The Data Type Changed and Broke Everything"
What you see: The pipeline crashes with "cannot cast VARCHAR to INTEGER" or "invalid date format". Data that used to load fine is now rejected at the transformation step.
🔌 Like trying to charge a UK phone in a US socket without an adapter — the plug shape changed, so it doesn't fit anymore.
✅ Fix: Add explicit type casting with error handling. Instead of assuming a value is a number, explicitly convert it (CAST) and handle cases where conversion fails — log them rather than crashing the whole pipeline.
⏳
Memory / Timeout — "The Pipeline Ran Out of Time or Space"
What you see: The pipeline times out or runs out of memory on a large table. It worked fine last month, but data volume has grown and now it can't finish in the time window.
🛒 Like trying to carry 100 shopping bags in one trip — the first 30 are fine, but eventually you physically can't carry more without making two trips.
✅ Fix: Add partitioning (process one day at a time instead of the whole table), increase the compute cluster size, or switch to micro-batching (many small loads instead of one giant one).
🕳️
Stale Incremental Load — "Some Records Got Silently Skipped"
What you see: The pipeline ran successfully, but some records are missing. Typically happens with incremental pipelines (those that only load "new" data since last run) where the logic for determining "what's new" is slightly off.
📰 Like a newspaper delivery that only picks up papers added after 6am — but some papers arrive at 5:55am and get left behind every single day.
✅ Fix: Add a small overlap window to your incremental logic (e.g. re-process the last 2 hours even if already processed). This catches records that arrived slightly outside the expected window. Monitor row counts daily to detect gaps early.

Non-Production Tables — "This Data Isn't Ready Yet"

Sometimes a table exists in the warehouse but isn't ready for business use — it has known issues, is under development, or needs sign-off. These must be clearly labelled, or teams will start relying on them without knowing the limitations.

🚫 Without a Label — What Goes Wrong
An analyst finds a new table called fct_returns_v2. It looks great! They build a Finance report on it. Nobody realised it was still in development and had duplicate order IDs. Three weeks later, Finance has made decisions based on wrong data.
✅ With Clear Labelling — What Happens Instead
The table shows a prominent "⚠ Not production-ready" banner in the data catalog with known issues listed. Nobody builds on it until the DQ team formally promotes it to production status with a passing check result attached.

For every non-production table, make sure all four of these are in place:

📁Data catalog warning — A visible "not production-ready" flag in dbt docs, Alation, DataHub, or whatever tool your team uses to browse tables
💬Code-level comment — A comment at the top of the dbt model or SQL file listing known issues and who owns the fix
📢Active communication — A Slack message or email to potential consumers: "This table exists but isn't ready yet — here's why and when it will be"
📋Formal promotion checklist — All DQ checks must pass and a named approver must sign off before the table moves from "under review" to "production"

Root Cause
Analysis Framework

🩺 Think of it like a doctor diagnosing a patient
When you go to a doctor with a headache, a good doctor doesn't just hand you paracetamol and send you home. They ask questions: How long? Where exactly? Any other symptoms? Recent changes in sleep or stress? They're not treating the headache — they're trying to find out what's causing it.

Root Cause Analysis (RCA) is the same mindset applied to data quality issues. Instead of just patching the broken column or re-running the pipeline (treating the symptom), you trace the problem all the way back to its origin — and fix the thing that caused it so it can't happen again.

Without RCA, you end up fixing the same issues over and over. With RCA, each incident makes your system permanently more resilient.

The Fishbone Diagram — Finding All the Possible Causes

The fishbone diagram (also called the Ishikawa diagram) is a visual tool for brainstorming all the possible causes of a problem. The "effect" (the DQ issue) sits at the right end. The "bones" branching off the spine represent different categories of cause. It prevents you from fixating on the first cause you think of and missing the real one.

For data quality, the six main cause categories are:

👥
People
  • → Training gaps — team didn't know the rule
  • → No ownership — nobody is responsible
  • → Process not followed consistently
📋
Process
  • → No data contract between teams
  • → Missing SLAs for data delivery
  • → No change notification process
💻
Systems
  • → Schema drift from source system
  • → Pipeline bug introduced by code change
  • → Infrastructure timeout or memory error
📡
Data Sources
  • → Bad data entered at origin
  • → Unexpected volume spike
  • → Source format changed silently
🛠️
Tools
  • → Wrong tool for the use case
  • → Tool upgrade breaking behaviour
  • → Inadequate monitoring coverage
🌍
Environment
  • → Cloud outage or network issue
  • → Environment config differences (dev vs prod)
  • → Regulatory or compliance change

The 6-Step RCA Framework — What to Do After a Serious Incident

For any P1 or P2 severity incident, follow these six steps in order. Don't skip to step 5 just because you think you know the answer — each earlier step is necessary to make sure you're solving the right problem:

1
🚨
Declare the Incident — Stop the Clock
As soon as a P1 or P2 is confirmed, formally declare it. This starts the response timer and ensures the right people are engaged. Write down: what failed, when it was first detected, and who is responsible for coordinating the response.
Example: "Incident declared at 02:17 UTC. Orders table missing 40% of today's rows. Response lead: on-call engineer. Business owner notified: Head of Operations."
2
📊
Assess the Impact — How Bad Is It?
Before you investigate the cause, understand the full blast radius. Which tables are affected? Which reports are wrong? Which business processes or customers are impacted? Quantify it in business terms — not just rows and percentages.
Example: "3 downstream tables affected: fct_orders, dim_order_history, rpt_daily_revenue. The revenue report due at 9am will be wrong by approximately £340,000. Fulfilment team is working from stale data."
3
🕰️
Reconstruct the Timeline — When Did Things Actually Go Wrong?
Build a minute-by-minute timeline: when was data last correct? When did the first signs appear? When was it detected? When did response start? Identifying the gaps — especially between "when it started" and "when we knew" — reveals weaknesses in your monitoring.
Example: "Data was correct at 23:50. Pipeline ran at 00:15. Error occurred silently at 00:22. Alert fired at 02:14. Response started at 02:17. Detection lag: 1h 52min — this needs to be reduced."
4
🔍
Apply Causal Analysis — Trace the Problem to Its Origin
Use the 5-Why technique (see Chapter 12) or the fishbone diagram to trace from symptom to root cause. Don't stop at the first "why" — keep going until you reach a systemic gap. Distinguish between the root cause (the fundamental reason) and contributing factors (things that made it worse).
Root cause vs contributing factor: Root cause — "no schema change notification process existed". Contributing factor — "the engineer was new and didn't know to check for dependencies". Only fixing the contributing factor means it'll happen again when the next new engineer joins.
5
🛠️
Define Corrective Actions — What Will Prevent This Forever?
Write specific, assigned, time-bound actions for three things: (1) fix the current issue, (2) prevent recurrence, and (3) improve detection speed if it happens again. Every action needs an owner and a deadline — actions without owners don't get done.
Example actions: (1) Fix the broken JOIN in the CRM pipeline — owner: Data Eng, by tomorrow. (2) Create a data contract with CRM team — owner: DQ Lead, by end of sprint. (3) Add schema drift alert to CRM pipeline — owner: Data Eng, by end of sprint.
6
📝
Write a Blameless Post-Mortem — So the Team Learns Together
A post-mortem is a written document shared with the whole team — and sometimes leadership — that captures everything learned from the incident. The key word is blameless: the goal is to understand what happened and improve the system, not to point fingers at individuals.
Blameless means: "Our monitoring wasn't sensitive enough to catch this quickly" — not "James didn't notice the alert". The first statement leads to a better monitoring system. The second leads to a scared, defensive team that hides problems.

The Post-Mortem Template — What to Document

Every P1/P2 incident should have a post-mortem written within 48 hours. Here's the standard structure — keep it short and factual:

📄 Post-Mortem Document Structure
Summary
One paragraph. What happened, when, and what the impact was. Written for a non-technical audience.
Timeline
Chronological log of events from the first sign of the problem to full resolution, with timestamps.
Root Cause
The single fundamental cause identified through 5-Why or fishbone analysis. Not the symptom — the systemic gap.
Impact
Which tables, reports, and business processes were affected. Business impact in pounds/dollars or customer count where possible.
What Went Well
Honest acknowledgement of things that worked — fast detection, good communication, quick resolution. Reinforce these behaviours.
Action Items
Specific improvements with named owners and deadlines. This is the most important section — it's where "never again" becomes real.
🌱 Building a Blameless Data Culture
The single biggest factor in whether a post-mortem is useful is whether people feel safe telling the truth. If team members fear being blamed or criticised, they will downplay what happened, omit details, and avoid acknowledging their role — making the document useless.

A blameless post-mortem assumes: Everyone involved made the best decisions they could with the information they had at the time. If a mistake was made, the system or process made it easy to make that mistake — and fixing the system is the goal.

Teams that run blameless post-mortems consistently have better data quality, faster incident resolution, and more psychological safety — because people flag problems early instead of hiding them until they become crises.

Stage 1 · Training Data — The 5 Things That Break AI Models

Bad training data creates bad models — and the problems don't become visible until the model is live and making wrong decisions. Here are the five most common training data quality failures, explained simply:

🏷️
Wrong Labels
Every training example must be correctly labelled. If 10% of "spam" emails are actually labelled "not spam", the model learns an incorrect definition of spam — and it will make the same mistakes forever.
🎓 Like a student who studied from an answer key that had 10% of the answers wrong. They'll pass practice tests incorrectly and never know.
⚠ Result: Model confidently gives wrong answers on the exact cases where labels were wrong
⚖️
Unbalanced Classes
If your training data has 99% "no fraud" examples and only 1% "fraud" examples, the model learns that the safest bet is to call everything "no fraud" — and it will be right 99% of the time, while missing every single actual fraud case.
🎲 Like training a coin-flip game on 10,000 heads and only 100 tails — the model just learns to always say "heads".
⚠ Result: High overall accuracy, but the model fails on exactly the cases that matter most
🔮
Data Leakage
Leakage happens when the model is accidentally given information during training that it won't have in real life — like using "loan_was_approved" as an input feature when predicting "should we approve this loan". The model cheats, and then fails in production.
📝 Like a student who memorised the exact exam questions in advance. They score 100% — but can't answer anything slightly different.
⚠ Result: Model looks perfect during testing, then fails completely when deployed
🌍
Unrepresentative Sample
If your training data only reflects certain customers, geographies, or time periods — the model won't know how to handle cases it's never seen. A model trained only on desktop users will make wrong predictions for mobile users.
🗺️ Like learning to drive only in the countryside — the model has no idea what to do in a city with traffic lights, pedestrians, and roundabouts.
⚠ Result: Model performs well for the majority it was trained on, badly for everyone else
📅
Temporal Integrity (Time Order)
For models that predict things over time (like "will this customer churn next month?"), the training data must be in the correct chronological order. Mixing up past and future data gives the model access to information it shouldn't have.
📰 Like teaching someone to predict tomorrow's news by accidentally including next week's newspapers in the "training pile".
⚠ Result: Artificially high accuracy in testing, then terrible predictions in the real world
🕳️
Missing Values
AI models can't handle missing data the way a human can (we just ignore the blank and use context). A missing value gets interpreted as a number (often 0 or -1) — which can wildly distort the model's understanding of that feature.
🔢 Like filling in every blank in a maths test with "0" — sometimes that's correct, but usually it's completely wrong and misleads all subsequent calculations.
⚠ Result: Model treats "unknown" as a meaningful value and makes predictions based on phantom patterns

Stage 2 · Feature Quality — What the Model Gets as Inputs Every Day

Even if your training data was perfect, the model still needs good inputs every time it makes a prediction. A trained model is like a formula — if you feed it wrong numbers, it gives wrong answers. Four things to check on every batch of features:

🚫
No Null Features
Every input column the model was trained on must be present for every prediction. A missing feature — even one — can cause wrong predictions or crashes.
✅ Check: All feature columns are populated. Alert if null rate on any feature exceeds 0.5%.
📏
Values in Valid Ranges
If the model was trained on customer lifetime values between £0 and £10,000, and it suddenly receives a value of £9,000,000 (a data entry error), the prediction will be nonsense.
✅ Check: All numeric features fall within expected min/max bounds from training distribution.
⏱️
Fresh Enough Data
"Days since last order" computed from yesterday's data is fine. Computed from data that's 3 weeks stale because a pipeline broke — the model is making predictions based on an outdated reality.
✅ Check: Feature store data is refreshed within the expected SLA window before any model scoring run.
📊
Stable Distributions
If the average customer age in training was 34 and today's batch average is 19, the model is seeing a completely different population. Its predictions are likely to be systematically wrong for this new group.
✅ Check: Mean and standard deviation of key features stay within 2 standard deviations of training baseline.

Stage 3 · Drift Detection — When the World Changes and the Model Doesn't Know

A model trained in January works well in February. By August, the world may have changed significantly — customer behaviour shifts, prices change, new products launch. But the model is still making decisions based on January's reality. This is called drift, and it's one of the sneakiest problems in AI.

📉 Data Drift — The Input Changes
The type of data coming into the model changes. The values shift. The distribution moves. The model was trained on one world, but it's now seeing a different one.
Example: A fraud detection model was trained when average order value was £45. A new product category launched and average orders are now £180. The model treats all large orders as suspicious — flagging legitimate purchases constantly.
🔀 Concept Drift — The Relationship Changes
The input data looks similar, but the meaning of that data has changed. The same pattern that used to mean X now means Y. The model's predictions become wrong even when its inputs look normal.
Example: During lockdown, "high home internet usage" became normal behaviour — not a sign of an unemployed person. A credit risk model trained before the pandemic wrongly penalised customers for this pattern for years.

Special Case: Data Quality for ChatGPT-style AI (LLMs)

What is an LLM? A Large Language Model (LLM) is the technology behind tools like ChatGPT, Gemini, and Claude. Instead of predicting a number (like a fraud score), it predicts text — generating answers, summaries, and conversations. Data quality affects LLMs in unique ways that traditional checks don't cover.

Many companies use LLMs with their own data via a technique called RAG (Retrieval-Augmented Generation) — the LLM looks up relevant documents from your database to help answer questions. The quality of those documents directly determines the quality of the answers.

👻
Hallucination
The model confidently states something that is completely made up — it sounds plausible but is factually wrong. This happens when the model doesn't have good information to draw from, so it "fills in the gaps" with invented details.
📚 Like asking a library assistant a question when all the relevant books are missing — instead of saying "I don't know", they make up a convincing-sounding answer.
Fix: Use RAG with a high-quality, regularly updated knowledge base. Validate outputs against known facts where possible.
📅
Knowledge Cutoff / Stale Data
An LLM's knowledge has an expiry date — it only knows about things that existed when it was trained. Ask it about last month's product launch and it'll either make something up or admit ignorance.
📰 Like asking someone who's been on a remote island for 2 years about current events — their knowledge is accurate for their time, but completely out of date now.
Fix: Use RAG to supplement the LLM with fresh, dated documents. Add freshness metadata so old documents are deprioritised automatically.
🔎
Poor Document Quality
In a RAG system, the LLM retrieves documents from your knowledge base before answering. If those documents are duplicated, contradictory, poorly written, or outdated — the AI's answers will reflect all of those flaws.
🗂️ Like searching a filing cabinet full of misfiled, duplicated, and half-finished documents — the answer you pull out is only as good as what's in the files.
Fix: Run DQ checks on your knowledge base: deduplication, freshness, completeness scoring. Only index documents that meet a quality threshold.
💉
Prompt Injection / Bad Content
Malicious or accidental bad content in your retrieved documents can manipulate the AI's behaviour. A document containing instructions like "ignore all previous rules and..." can override the model's intended behaviour.
🔧 Like a saboteur sneaking instructions into a recipe book — the chef follows them faithfully, not knowing those instructions weren't supposed to be there.
Fix: Add content filtering to your document ingestion pipeline. Sanitise inputs and validate that retrieved content matches expected format and topic.

Data Governance &
Compliance Framework

🏢 Think of governance like the rules of a shared office kitchen
In a shared kitchen, without rules, things go wrong fast — someone's food gets eaten, dishes pile up, nobody labels anything. So you establish rules: label your food, clean up after yourself, don't leave dishes in the sink. Those rules are governance.

Data governance works the same way — it's the agreed set of rules, roles, and processes that determine how an organisation handles data responsibly. Data quality is one pillar of governance, but governance also covers: who owns what data, what the rules are for using it, how long it's kept, who can access it, and what happens when something goes wrong.

Without governance, teams make conflicting decisions about the same data, quality problems repeat indefinitely, and nobody is accountable. With governance, data becomes a managed, trustworthy asset.

The 5 Layers of Data Governance — Built from the Bottom Up

Governance isn't built in one go — it's layered. Each layer depends on the one below it. You can't have a strong quality monitoring programme (Layer 4) if nobody owns the data (Layer 1). Build from the bottom up:

1
🏛️ Governance Organisation — Who's Responsible for What
Defines the people and structures: a Data Council sets strategy, Data Owners are accountable for specific datasets, Data Stewards handle day-to-day quality. Without this layer, nobody is responsible — so nothing gets fixed.
2
📋 Policies & Standards — The Agreed Rules
Data classification (what's sensitive vs. public), retention policies (how long to keep data), access controls (who can see what), and naming conventions. These are the "office kitchen rules" — written down and applied consistently.
3
🗂️ Metadata & Catalog — Your Data's Address Book
A data catalog tells everyone what data exists, where it lives, who owns it, and what it means. Lineage tracking shows where data came from and where it flows. Without this, teams don't know what they have — and can't trust what they find.
4
🔍 Quality & Monitoring — Automated Checks Running Every Day
Automated DQ checks, KPI dashboards, incident management, and exception handling — the entire programme described in this handbook. This layer can only work if Layers 1–3 are in place: you need owners to escalate to, standards to measure against, and a catalog to know what to check.
5
🌱 Culture & Literacy — Everyone Cares About Data
Training, recognition, and embedding data quality into how teams work. This is the hardest layer to build — but it's what makes everything else sustainable. When engineers add DQ tests without being asked, and business leaders ask about data scores in reviews, you've built a data culture.

Data Contracts — A Formal Handshake Between Teams

A data contract is a written agreement between the team that produces data (e.g. the CRM system team) and the team that consumes it (e.g. the analytics team). It defines exactly what the producer will deliver, what quality it will meet, and what happens if they need to change it.

Think of it like a supplier agreement: "You'll deliver 500 units every Monday by 8am, all meeting spec X. If anything changes, you'll give us 2 weeks' notice." Data contracts prevent the silent changes that cause DQ failures — a column gets renamed, a field starts arriving null, a table stops updating — all without anyone telling the consumers.

📐
Schema Specification
Agreed column names, data types, and which fields must never be empty. Any change requires advance notice.
📊
Quality SLAs
Minimum completeness, freshness, and validity thresholds the producer commits to maintaining.
⏰
Delivery SLA
Commitment on when data will be available each day (e.g. by 06:00 UTC). Feeds downstream pipeline scheduling.
🔔
Change Management
Notice period (e.g. 2 weeks) and process for any schema, volume, or format changes. No silent changes allowed.
👤
Owner Contact
Named person on the producer side to escalate DQ issues to — not just a team inbox that nobody reads.
📜
Version History
A log of all changes to the contract over time — so you know what was agreed when, and can trace issues back to contract changes.

dbt Best Practices

dbt is the most popular tool for data transformation and testing. Here are 6 simple habits that separate good dbt projects from great ones:

1
Test every ID column

Add not_null and unique tests to every primary key and foreign key. IDs are the backbone of joins — one bad ID can corrupt an entire dashboard.

2
Write custom tests for business rules

Not every rule fits a built-in test. If "order total must never be negative" matters to your business, write a test for it. Business logic belongs in code, not in Slack messages.

3
Use severity levels

Mark critical checks as error (pipeline stops) and soft warnings as warn (pipeline continues but flags it). Not every DQ issue is an emergency.

4
Document every model

A future teammate (or future you) will thank you. Write a plain-English description for every table and column. dbt turns these into a browsable website automatically.

5
Add dbt-expectations for advanced checks

The dbt-expectations package adds 50+ powerful checks like "value must be between X and Y" or "column must match regex pattern." One package install, no custom SQL needed.

6
Track history with Elementary

Elementary sits inside your dbt project and records every test result over time. Instead of just knowing a test failed today, you can see it started failing 3 weeks ago — and pinpoint what changed.
